Queues (ACD)

Automatic Call Distribution queues with agent tiers, hold music, position announcements, wrap-up time, overflow routing, and real-time statistics.

Overview

A queue is a call center feature that holds incoming callers in line while the system finds the next available agent to handle the call. Callers hear music on hold and periodic announcements (position in queue, estimated wait time) while they wait. Agents are selected based on the queue's distribution strategy, their tier level, and their current availability status.

Queues are the right choice when you need formal call center capabilities: agent login/logout, idle-time-based distribution, tiered escalation, wrap-up time between calls, real-time statistics, and overflow routing. For simpler scenarios where you just need multiple phones to ring, see Ring Groups.

Queue vs Ring Group
Queues provide hold music, position announcements, agent status management, tiered routing, wrap-up time, and detailed statistics. Ring groups simply ring a set of phones. Use queues for customer-facing support and sales lines; use ring groups for internal teams.

Queue Call Flow

The following diagram shows the complete lifecycle of a call entering a queue.

Caller → Queue │ ├─ No agents logged in? │ └─ Wait max_wait_time_with_no_agent (120s) │ └─ Still no agents → Overflow destination │ ├─ In queue (MOH plays, announcements every announce_frequency) │ │ │ ├─ Agent becomes available │ │ └─ Ring agent (ring_timeout = 30s) │ │ ├─ Agent answers → Connected (record if enabled) │ │ │ └─ Call ends → Wrap-up time → Agent available │ │ └─ No answer → Next agent (per strategy) │ │ │ ├─ max_wait_time reached (300s) │ │ └─ Overflow destination │ │ │ └─ Caller presses caller_exit_key │ └─ Exit action (overflow destination) │ └─ Caller hangs up ├─ Within discard_abandoned_after? → Marked abandoned └─ Caller returns + abandoned_resume_allowed? → Resume original position

Queue Strategies

The queue strategy determines how the next available agent is selected when a caller is ready to be connected. All strategies only consider agents whose status is AVAILABLE.

Strategy Selection Logic Best For
LONGEST_IDLE_AGENT Selects the agent who has been idle (not on a call) the longest. Ensures even workload distribution based on actual availability. Most call centers. This is the default and the most commonly used strategy.
ROUND_ROBIN Cycles through agents in a fixed order, starting from where the last call left off. Each agent gets one call before the cycle repeats. Teams where strict rotation is preferred regardless of idle time.
RANDOM Picks a random available agent. Simple but may lead to uneven distribution with small teams. Large agent pools where statistical distribution is sufficient.
FEWEST_CALLS Selects the agent who has handled the fewest calls in the current period. Balances total call count rather than idle time. Environments where total call count fairness matters more than idle time (e.g., sales with commission).
WEIGHTED Uses agent position values as weights. Higher-positioned agents receive a proportionally larger share of calls. Tiered support where senior agents should handle more calls, or skills-based routing approximation.

Creating a Queue

Submit a POST request with the queue configuration and an initial list of members. At minimum, provide a name and queue number. 

POST /api/v1/queues

{
  "name": "Technical Support",
  "queue_number": "800",
  "description": "Tier 1 technical support queue",
  "strategy": "LONGEST_IDLE_AGENT",
  "max_wait_time": 300,
  "max_wait_time_with_no_agent": 120,
  "tier_rules_apply": true,
  "tier_rule_wait_second": 300,
  "tier_rule_wait_multiply_level": true,
  "wrapup_time": 15,
  "ring_timeout": 30,
  "announce_frequency": 60,
  "caller_exit_key": "0",
  "overflow_destination_type": "VOICEMAIL",
  "overflow_destination_id": "aaaabbbb-cccc-dddd-eeee-ffffffffffff",
  "record_calls": true,
  "enabled": true,
  "members": [
    { "extension_id": "ext-1-uuid", "level": 1, "position": 1 },
    { "extension_id": "ext-2-uuid", "level": 1, "position": 2 },
    { "extension_id": "ext-3-uuid", "level": 2, "position": 1 }
  ]
}

Field Reference

QueueCreate Fields

Field Type Required Default Description
name string Yes — Descriptive name for the queue (1–255 characters)
queue_number string Yes — Dial number to reach the queue (1–20 characters). Example: "800"
description string No null Optional description of the queue's purpose
strategy enum No LONGEST_IDLE_AGENT Agent selection strategy: LONGEST_IDLE_AGENT, ROUND_ROBIN, RANDOM, FEWEST_CALLS, WEIGHTED
moh_prompt_id UUID No null Audio prompt for music on hold. Falls back to system default if not set.
max_wait_time int No 300 Maximum seconds a caller waits in queue before overflow (0–7200). Set to 0 for unlimited wait.
max_wait_time_with_no_agent int No 120 Maximum seconds a caller waits when no agents are logged in (0–7200). Typically shorter than max_wait_time since no agent is expected.
tier_rules_apply boolean No true Whether tier-based escalation is active. See Tier Rules.
tier_rule_wait_second int No 300 Seconds to wait at each tier before escalating to the next (0–7200)
tier_rule_wait_multiply_level boolean No true When true, the wait time is multiplied by the tier level. Tier 1 waits tier_rule_wait_second, tier 2 waits 2 × tier_rule_wait_second, etc.
tier_rule_no_agent_no_wait boolean No false When true, if no agents are available at the current tier, immediately escalate to the next tier without waiting.
discard_abandoned_after int No 60 Seconds after which an abandoned call's position is discarded (0–7200). Only relevant when abandoned_resume_allowed is true.
abandoned_resume_allowed boolean No false Allow callers who hung up and called back to resume their original position in the queue
caller_exit_key string No null DTMF key that allows the caller to exit the queue (max 5 characters). Common values: "0", "#". The caller is sent to the overflow destination.
wrapup_time int No 0 Seconds after a call ends before the agent receives the next call (0–600). Gives agents time to complete notes.
ring_timeout int No 30 Seconds to ring an agent before considering it unanswered and moving to the next agent (5–300)
announce_frequency int No 0 Seconds between position/wait-time announcements to the caller (0–600). Set to 0 to disable announcements.
announce_prompt_id UUID No null Custom audio prompt for announcements. Falls back to system default if not set.
overflow_destination_type string No null Destination type when the queue overflows (max 20 characters). Same types as inbound routes.
overflow_destination_id UUID No null UUID of the overflow destination object
record_calls boolean No false Automatically record all calls handled by this queue. Recordings are stored in the recordings system.
enabled boolean No true Whether the queue is active
members list No [] List of QueueMemberCreate objects. See Agent/Member Management.

Agent / Member Management

Queue members are the agents (extensions) that handle calls from the queue. Each member has a tier level and a position within that tier. The combination of level and position determines when and in what order an agent is considered for calls.

Member Structure

  • Level — The tier the agent belongs to (1 = highest priority). Calls are offered to level 1 agents first. If no level 1 agents are available and tier rules allow escalation, level 2 agents are tried, and so on.
  • Position — The agent's order within their tier. Used by strategies like round-robin and weighted to determine sequence. Lower position numbers are tried first.

Example: Two-Tier Support Team

"members": [
  { "extension_id": "agent-a", "level": 1, "position": 1 },
  { "extension_id": "agent-b", "level": 1, "position": 2 },
  { "extension_id": "agent-c", "level": 1, "position": 3 },
  { "extension_id": "senior-d", "level": 2, "position": 1 },
  { "extension_id": "senior-e", "level": 2, "position": 2 }
]

In this setup, agents A, B, and C handle calls first (tier 1). If all tier 1 agents are busy and the tier escalation timer expires, calls are offered to senior agents D and E (tier 2).

Tier Rules

Tier rules control how and when calls escalate from one agent tier to the next. These settings only apply when tier_rules_apply is true.

Escalation Behavior

Setting Effect
tier_rule_wait_second: 300 Wait 300 seconds (5 minutes) at the current tier before escalating to the next tier.
tier_rule_wait_multiply_level: true Tier 1 waits 300s, tier 2 waits 600s, tier 3 waits 900s, etc. This gives higher tiers more time because they handle overflow.
tier_rule_wait_multiply_level: false Every tier waits exactly tier_rule_wait_second before escalating.
tier_rule_no_agent_no_wait: true If no agents at the current tier are available (all busy, offline, or on break), immediately skip to the next tier without waiting.
tier_rule_no_agent_no_wait: false Wait the full timer even if no agents are available at the current tier. An agent may become available during the wait.
Tier escalation example (tier_rule_wait_second: 120, multiply: true) 0:00 → Try Tier 1 agents (level: 1) 2:00 → No answer from Tier 1 → Escalate to Tier 2 (level: 2) 6:00 → No answer from Tier 2 → Escalate to Tier 3 (level: 3) 12:00 → No answer from Tier 3 → All tiers exhausted → max_wait_time still running? Wait for any agent. → max_wait_time expired? Overflow destination.
Disabling Tiers
Set tier_rules_apply: false to treat all members as a single pool regardless of their level value. The queue strategy will select from all available agents without tier-based ordering.

Caller Experience

Several settings shape what the caller hears and can do while waiting in the queue.

Music on Hold

The moh_prompt_id field specifies an audio file played continuously while the caller waits. If not set, the system default music is used. Upload custom audio files via Audio Prompts.

Position and Wait Time Announcements

When announce_frequency is greater than 0, the caller periodically hears their position in the queue and estimated wait time. For example, "You are caller number 3. Estimated wait time is 4 minutes." The announcement interrupts music on hold briefly, then MOH resumes.

  • announce_frequency: 60 — Announce every 60 seconds
  • announce_frequency: 0 — No announcements (caller only hears MOH)
  • announce_prompt_id — Custom announcement audio (optional)

Caller Exit Key

The caller_exit_key allows callers to press a DTMF key to leave the queue voluntarily. The call is then sent to the overflow destination. This is typically used to let callers leave a voicemail instead of waiting.

  • "0" — Press 0 to leave the queue (common for "press 0 to leave a message")
  • "#" — Press # to exit
  • null — No exit key (caller must wait or hang up)

Abandoned Call Resume

When abandoned_resume_allowed is true, a caller who hangs up and calls back within discard_abandoned_after seconds can resume their original position in the queue instead of going to the back of the line. The system matches by caller ID.

Overflow and Failover

The overflow destination receives calls when the queue cannot serve them. Overflow triggers include:

  • Max wait time exceeded — Caller has been in the queue longer than max_wait_time seconds.
  • No agents available — No agents are logged in and max_wait_time_with_no_agent has expired.
  • Caller pressed exit key — Caller pressed caller_exit_key to leave voluntarily.

If no overflow destination is configured, the caller hears a busy signal when overflow conditions are met. Always configure an overflow destination (typically a voicemail box or an announcement) for a professional caller experience.

Wrap-Up Time

The wrapup_time setting adds a pause after each queue call before the agent is offered the next call. During wrap-up, the agent's status remains BUSY from the queue's perspective, giving them time to complete call notes, update tickets, or take a breath.

Value Behavior
0 (default) No wrap-up. The agent is immediately available for the next call.
15 15-second pause after each call. Good for quick note-taking.
60 60-second pause. Suitable for calls requiring ticket updates.
300 5-minute pause. For complex calls requiring detailed follow-up.

Agent Status Management

Each agent has a status that determines whether they receive queue calls. Agents can change their own status (within their permissions), and supervisors can change any agent's status.

Status Receives Calls Description
AVAILABLE Yes Agent is logged in and ready to take calls. This is the only status that receives queue calls.
BUSY No Agent is currently on a call or in wrap-up. Set automatically by the system; returns to AVAILABLE when the call (and wrap-up) ends.
DO_NOT_DISTURB No Agent has manually entered DND mode. No queue calls are offered. The agent must manually return to AVAILABLE.
ON_BREAK No Agent is on a scheduled or manual break. Similar to DND but tracked separately for reporting.
OFFLINE No Agent is logged out of the queue. They do not appear in the available agent pool.
Status vs Queue Membership
Agent status is separate from queue membership. An agent can be a member of a queue (configured in the members list) but have a status of OFFLINE, meaning they are not currently taking calls from that queue. Removing an agent from the members list removes them permanently; changing status to OFFLINE is temporary.

Queue Statistics

The platform provides real-time queue statistics through the QueueStatsResponse object. These metrics are available via the API and displayed on the Dashboard.

QueueStatsResponse Fields

Field Type Description
queue_id UUID The queue these stats belong to
queue_name string Display name of the queue
waiting_count int Number of callers currently waiting in the queue
agents_logged_in int Total number of agents with any status other than OFFLINE
agents_available int Number of agents with status AVAILABLE (ready to take calls)
agents_on_call int Number of agents currently handling a queue call
longest_wait_seconds int How long the longest-waiting caller has been in the queue (seconds) 
GET /api/v1/queues/{queue_id}/stats

{
  "queue_id": "aaaabbbb-cccc-dddd-eeee-ffffffffffff",
  "queue_name": "Technical Support",
  "waiting_count": 3,
  "agents_logged_in": 5,
  "agents_available": 2,
  "agents_on_call": 3,
  "longest_wait_seconds": 127
}

Call Recording in Queues

When record_calls is true, every call handled by the queue is automatically recorded from the moment the agent answers until the call ends. The recording captures both sides of the conversation.

  • Recordings are stored in the platform's object storage (MinIO/S3)
  • Each recording is linked to the CDR (call detail record) for the call
  • Recordings can be played back and downloaded from Recordings
  • Storage retention follows the tenant's configured retention policy
Legal Compliance
Call recording laws vary by jurisdiction. Some states and countries require all-party consent (all participants must be informed). Configure an announcement prompt that notifies callers their call may be recorded. Consult legal counsel for your specific requirements.
