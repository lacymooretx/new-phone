Conference Bridges

Set up multi-party conference rooms with moderator controls, PIN protection, join/leave announcements, and optional recording.

Overview

Conference bridges allow multiple participants to join a shared audio room by dialing a room number. Each bridge operates independently with its own settings for capacity, PIN protection, moderation, and recording. Conferences are ideal for recurring team meetings, customer calls with multiple stakeholders, and ad-hoc collaboration.

The platform distinguishes between two participant roles: moderators who have elevated control and participants who join as regular attendees. A moderator PIN grants moderator privileges; a participant PIN (if set) simply restricts access to authorized users.

Dialing In
Participants join a conference by dialing the room_number from any registered extension, the WebRTC softphone, or (if routed via an inbound route) from an external phone number.

Creating a Conference Bridge

Create a conference bridge by submitting a POST request with the room configuration. The only required fields are name and room_number. 

POST /api/v1/conferences

{
  "name": "Weekly Engineering Standup",
  "room_number": "800",
  "description": "Every Monday 9:00 AM team standup",
  "max_participants": 25,
  "participant_pin": "1234",
  "moderator_pin": "5678",
  "wait_for_moderator": true,
  "announce_join_leave": true,
  "moh_prompt_id": null,
  "record_conference": false,
  "muted_on_join": false,
  "enabled": true
}

PIN Protection

Conferences support two distinct PINs that serve different purposes. PINs are entirely optional — omit both for an open conference room that anyone with the room number can join.

Participant PIN

When participant_pin is set, callers must enter the correct PIN before being admitted to the conference. This restricts access to authorized attendees only. The system prompts the caller to enter the PIN and grants up to three attempts before disconnecting.

Moderator PIN

When moderator_pin is set, callers who enter this PIN instead of the participant PIN are granted moderator privileges. Moderators are never muted on join (even when muted_on_join is true) and their presence satisfies the wait_for_moderator requirement.

PIN Security
Use different values for participant and moderator PINs. If both PINs are the same, every caller receives moderator privileges. For sensitive meetings, use PINs of at least 6 digits and change them periodically.

PIN Behavior Matrix

Participant PIN Moderator PIN Behavior
Not set Not set Open access. Anyone who dials the room number joins immediately as a regular participant.
Set Not set PIN required for entry. All callers enter as regular participants. No moderator privileges available.
Not set Set Open access for participants. Callers who enter the moderator PIN receive moderator privileges.
Set Set All callers must enter a PIN. The system checks against both PINs to determine the caller's role.

Moderator Features

Moderators have elevated control over the conference session. Their presence can gate whether other participants can hear each other, and they bypass the mute-on-join setting.

Wait for Moderator

When wait_for_moderator is true, participants who join before any moderator are placed on hold and hear music on hold (from moh_prompt_id or the system default). They cannot hear each other. Once a moderator joins, all waiting participants are connected into the conference simultaneously.

If the last moderator leaves the conference while participants remain, the behavior depends on your deployment configuration. By default, participants remain connected but new callers will again be placed on hold until a moderator rejoins.

Mute on Join

When muted_on_join is true, all participants join the conference with their microphone muted. This is useful for large meetings, town halls, or presentations where only the moderator should speak initially. Participants can unmute themselves using their phone's mute toggle or a DTMF command (if supported by the handset). Moderators are never muted on join.

Join and Leave Announcements

When announce_join_leave is true, existing participants hear a notification tone each time someone joins or leaves the conference. This helps attendees stay aware of who is present without checking a participant list.

  • Join tone — A brief chime played to all current participants when a new caller enters.
  • Leave tone — A different tone played when a participant disconnects.

For large conferences (20+ participants), consider disabling announcements to avoid constant notification sounds disrupting the meeting. For small team meetings, announcements provide helpful context about attendance.

Recording Conferences

When record_conference is true, the entire conference session is recorded from the moment the first participant joins until the last participant leaves. The recording captures a mixed audio stream of all participants.

  • Recordings are stored in the platform's object storage (MinIO/S3)
  • Each recording is linked to the conference session and accessible from Recordings
  • Storage retention follows the tenant's configured retention policy
  • Recordings can be downloaded for archival or compliance purposes
Recording Compliance
Recording laws vary by jurisdiction. Some require all-party consent, meaning every participant must be informed the conference is being recorded. Consider playing an announcement at the start of recorded conferences. Consult legal counsel for your specific requirements.

Capacity Planning

The max_participants field controls the maximum number of simultaneous participants in a single conference room. The platform supports up to 300 participants per conference, though real-world capacity depends on your server resources.

Participants Use Case Notes
5–10 Team standups, small meetings Minimal resource usage. Announcements and recording work well.
10–50 Department meetings, project syncs Consider muted_on_join to reduce background noise.
50–150 Company-wide meetings, town halls Use muted_on_join: true and disable announce_join_leave.
150–300 Large events, webinars Monitor server CPU and bandwidth. Recording increases resource usage. Consider a dedicated media server.

Setting max_participants: 0 removes the participant limit entirely. Use this only when you are confident in your server capacity, as an unlimited conference can consume significant CPU and bandwidth.

Common Use Cases

Recurring Team Meeting

Create a dedicated conference room for each team with a fixed room number. Team members dial the same number every day. Use wait_for_moderator: false so the first person to dial in can start talking immediately. 

{
  "name": "Dev Team Daily",
  "room_number": "801",
  "max_participants": 15,
  "wait_for_moderator": false,
  "announce_join_leave": true,
  "muted_on_join": false
}

Customer Conference Call

Secure conferences with external participants. Set both PINs so only invited callers can join, and enable recording for documentation. Route an external DID to the conference via an inbound route so external callers can dial in. 

{
  "name": "Client Review - Acme Corp",
  "room_number": "850",
  "max_participants": 20,
  "participant_pin": "924731",
  "moderator_pin": "186402",
  "wait_for_moderator": true,
  "record_conference": true
}

Company Town Hall

Large presentation-style conferences. Mute everyone on join so only the moderator (presenter) speaks. Disable join/leave sounds to avoid constant notifications. Record for employees who could not attend. 

{
  "name": "Q1 All-Hands",
  "room_number": "899",
  "max_participants": 200,
  "moderator_pin": "770055",
  "wait_for_moderator": true,
  "announce_join_leave": false,
  "record_conference": true,
  "muted_on_join": true
}
