Roles & Permissions

Role-Based Access Control (RBAC) defines what each user can see and do across the platform.

Overview

The Aspendora Connect uses a five-role RBAC model that separates MSP-level operations from tenant-level management. Each user is assigned exactly one role, and that role determines their complete set of permissions. Permissions are enforced at the API level on every request, ensuring consistent access control regardless of client.

Roles are hierarchical in terms of capability but not inheritance — each role has an explicitly defined set of permissions. There is no implicit "everything the lower role can do" logic; instead, the full permission set is enumerated for each role.

Tenant Isolation
RBAC works in conjunction with PostgreSQL Row-Level Security (RLS). Even if a permission is granted, a tenant-scoped user can only access data belonging to their own tenant. MSP roles can access data across all tenants.

Role Descriptions

MSP Super Admin msp_super_admin

Full platform access with no restrictions. This is the highest-privilege role, intended for the MSP organization's top-level administrators. MSP Super Admins can manage all tenants, all settings, and all features across the entire platform. This is the only role with the manage_platform permission, which controls global platform settings such as system configuration, license management, and platform-wide policies.

  • Create, modify, and delete tenants
  • Manage platform-wide settings and configurations
  • Access all data across all tenants
  • Assign any role to any user
  • Full access to audit logs across the platform

MSP Tech msp_tech

Nearly identical to MSP Super Admin but without the manage_platform permission. Intended for MSP technicians who need cross-tenant access for support and configuration tasks but should not modify global platform settings.

  • Access and manage all tenants
  • Configure telephony, connectivity, and system settings for any tenant
  • Manage users and roles within any tenant
  • Cannot modify global platform configuration

Tenant Admin tenant_admin

Full access within their assigned tenant. Tenant Admins can manage all features, users, and settings for their organization. They cannot see other tenants or access platform-level settings.

  • Manage all telephony features (extensions, trunks, routes, queues, IVR, etc.)
  • Create and manage users within their tenant
  • View audit logs for their tenant
  • Configure DIDs, inbound/outbound routes, and SIP trunks
  • Cannot access other tenants or platform settings

Tenant Manager tenant_manager

Mid-level access within a tenant. Managers can handle day-to-day user and extension management, view most configurations, and manage recordings and voicemail. They cannot manage trunks, DIDs, routes, IVR menus, queues, conferences, or paging groups.

  • Manage users and extensions
  • Manage ring groups, recordings, voicemail, and voicemail messages
  • View (but not modify) trunks, DIDs, routes, IVR, queues, conferences, and paging
  • View call history and audit logs
  • Cannot create or modify connectivity settings

Tenant User tenant_user

Basic end-user access. Tenant Users can view their own profile, see extensions and ring groups, access voicemail, view call history, and listen to recordings. They have no management capabilities.

  • View and edit own profile
  • View extensions and ring groups
  • Access voicemail and voicemail messages (view only)
  • View call history and recordings
  • No management or configuration access

Role Comparison Summary

A quick reference for understanding the scope and boundaries of each role.

Role Scope Key Capability Key Limitation
msp_super_admin All tenants + platform Full unrestricted access; manage_platform None
msp_tech All tenants Cross-tenant support and configuration No manage_platform
tenant_admin Single tenant Full tenant management including connectivity No cross-tenant access; no platform settings
tenant_manager Single tenant User/extension management; recordings; voicemail No trunks, DIDs, routes, IVR, queues, conferences, paging management
tenant_user Own data only View own profile, extensions, voicemail, CDRs No management capabilities; view-only

Assigning Roles

Roles are assigned when creating or editing a user account. The role you can assign depends on your own role:

  • MSP Super Admin — Can assign any role to any user in any tenant
  • MSP Tech — Can assign any tenant-level role in any tenant (cannot create MSP-level users)
  • Tenant Admin — Can assign tenant_admin, tenant_manager, or tenant_user within their tenant
  • Tenant Manager — Can assign tenant_manager or tenant_user within their tenant
  • Tenant User — Cannot assign roles
Role Escalation Prevention
Users cannot assign a role with higher privileges than their own. A tenant_manager cannot promote a user to tenant_admin. This is enforced at the API level.

Permission Enforcement

Permissions are enforced on every API request through middleware that inspects the JWT access token. The enforcement flow is:

  1. Extract the JWT from the Authorization: Bearer header
  2. Verify the token signature and check expiration
  3. Extract role and tenant_id from token claims
  4. Look up the role's permission set
  5. Check that the required permission for the endpoint is in the set
  6. Set the app.current_tenant context for PostgreSQL RLS
  7. If all checks pass, execute the request; otherwise return 403 Forbidden
Defense in Depth
Even if a permission check is somehow bypassed, PostgreSQL RLS provides a second layer of tenant isolation at the database level. Tenant-scoped users can never read or write data belonging to another tenant, regardless of API-level checks.
