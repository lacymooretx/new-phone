Outbound Routes

Define dial patterns, digit manipulation, caller ID policy, and trunk selection for outgoing calls.

Overview

An outbound route controls how calls placed by users reach the outside world. When a user dials a number, the platform evaluates the dialed digits against all configured outbound routes, selects the best match based on dial pattern and priority, manipulates the digits as needed, sets the outbound caller ID, and sends the call out through the assigned SIP trunk. If the primary trunk is unavailable, the call automatically fails over to the next trunk in the ordered list.

Without outbound routes, users cannot make external calls. You need at least one route with a pattern that matches the numbers your users need to dial.

Dial Pattern Matching Flow

The following diagram shows how the outbound routing engine processes a dialed number.

User dials: 15551234567 │ ▼ Evaluate all enabled outbound routes (sorted by priority ASC) │ ├─ Route "Emergency" (priority 1) │ pattern: ^911$ → NO MATCH → skip │ ├─ Route "Long Distance" (priority 50) │ pattern: ^1[0-9]{10}$ → MATCH! │ ├─ Strip 0 digits: 15551234567 │ ├─ Prepend nothing: 15551234567 │ ├─ CID mode: EXTENSION → use ext CID │ └─ Send via trunk_ids[0] │ ├─ Trunk 1 available? → Yes → Call placed │ └─ Trunk 1 down? → Try trunk_ids[1] → ... │ ├─ Route "International" (priority 100) │ pattern: ^011 → not evaluated (already matched) │ └─ No match → Call rejected ("no route to destination")

Creating an Outbound Route

Submit a POST request to the outbound routes endpoint. You must provide a name, a dial pattern (regex), and at least one trunk ID. All other fields have sensible defaults. 

POST /api/v1/outbound-routes

{
  "name": "US Long Distance",
  "dial_pattern": "^1[0-9]{10}$",
  "prepend_digits": "",
  "strip_digits": 0,
  "cid_mode": "EXTENSION",
  "priority": 50,
  "enabled": true,
  "trunk_ids": [
    "aaaabbbb-cccc-dddd-eeee-ffffffffffff",
    "11112222-3333-4444-5555-666677778888"
  ]
}

Field Reference

OutboundRouteCreate Fields

Field Type Required Default Description
name string Yes — Descriptive name for the route (1–255 characters)
dial_pattern string Yes — Regular expression matched against the dialed number (1–100 characters). See Dial Pattern Syntax.
prepend_digits string No "" Digits added to the front of the number after stripping (max 20 characters). Used to add country codes or trunk prefixes.
strip_digits int No 0 Number of digits to remove from the front of the dialed number before prepending (0–20).
cid_mode enum No EXTENSION How outbound caller ID is determined. One of: EXTENSION, CUSTOM, DID. See CID Modes.
custom_cid string No null E.164 phone number used as caller ID when cid_mode is CUSTOM (max 40 characters). Example: "+15551234567".
priority int No 100 Route evaluation order. Lower numbers are evaluated first (0–9999). Routes with the same priority are evaluated in creation order.
enabled boolean No true Whether the route is active. Disabled routes are skipped during dial plan evaluation.
trunk_ids list[UUID] No [] Ordered list of SIP trunk IDs. The first trunk is the primary; subsequent trunks are failover in order. At least one trunk should be assigned for the route to work.

Response Fields

The API returns all create fields plus the following system-managed fields.

Field Type Description
id UUID Unique identifier for the outbound route
tenant_id UUID The tenant that owns this route
is_active boolean Computed field reflecting whether the route is enabled and has at least one trunk assigned
created_at datetime Timestamp when the route was created
updated_at datetime Timestamp of the last modification

Dial Pattern Syntax

Dial patterns use standard regular expression syntax. The pattern is matched against the complete dialed string. Use anchors (^ and $) to ensure exact matching, or omit $ to allow prefix matching.

Quick Regex Reference

Symbol Meaning Example
^ Start of string ^1 — must start with 1
$ End of string 567$ — must end with 567
[0-9] Any single digit [0-9]{3} — exactly 3 digits
{n} Exactly n repetitions [0-9]{10} — exactly 10 digits
{n,m} Between n and m repetitions [0-9]{7,11} — 7 to 11 digits
? Zero or one of preceding 1? — optional leading 1
. Any single character .* — anything

Common Dial Patterns

Name Pattern Matches Notes
Emergency ^911$ 911 Priority 1. Must always be the highest priority route.
Local (7-digit) ^[0-9]{7}$ 5551234 Some regions still use 7-digit local dialing.
Local (10-digit) ^[0-9]{10}$ 5551234567 10-digit dialing without leading 1.
Long Distance (11-digit) ^1[0-9]{10}$ 15551234567 US/Canada long distance with leading 1.
Flexible US/Canada ^1?[0-9]{10}$ 5551234567 or 15551234567 Matches with or without leading 1.
International ^011 01144207946xxxx International dialing prefix (011). No $ anchor allows variable length.
Toll-free ^1?(800|888|877|866|855|844|833)[0-9]{7}$ 18005551234 US toll-free numbers.
N11 codes ^[2-9]11$ 411, 511, 811 Information and service codes (not 911, which has its own route).
Always Create an Emergency Route
Every tenant must have an emergency route for ^911$ (and optionally ^933$ for E911 testing) at priority 1. This ensures emergency calls are never blocked by misconfigured dial plans.

Digit Manipulation

After a dialed number matches a route, the platform can modify the digits before sending them to the trunk. This is done in two steps, applied in order:

  1. Strip — Remove strip_digits characters from the front of the dialed number.
  2. Prepend — Add prepend_digits to the front of the remaining number.

Digit Manipulation Examples

Dialed Strip Prepend Sent to Trunk Use Case
95551234567 1 1 15551234567 Users dial 9 for outside line; strip the 9, prepend country code 1
15551234567 1 — 5551234567 Trunk expects 10 digits; strip leading 1
5551234 0 1555 15555551234 7-digit local dialing; prepend country code + area code
01144207946 3 + +44207946 Strip international prefix 011, prepend + for E.164

Caller ID Modes

The cid_mode field determines what phone number is shown to the party being called. Carriers may override this based on their own policies, but the platform sends the configured value.

Mode Behavior When to Use
EXTENSION Uses the outbound caller ID configured on the originating extension. Each user can have their own CID. Default choice. Users who have their own DID and want their direct number displayed.
CUSTOM Uses the E.164 number specified in custom_cid for all calls on this route. When all calls on a route must display the same number. Example: a company main number on all outbound calls.
DID Uses the DID assigned to the extension. Falls back to extension CID if no DID is assigned. When DID assignment drives caller ID rather than manual extension configuration.
Carrier CID Enforcement
Some SIP trunk providers enforce STIR/SHAKEN attestation and will only allow caller ID numbers that you own on that trunk. Sending an unauthorized CID may result in the call being rejected or displayed as "Spam Likely." Verify your CID configuration with your trunk provider.

Priority and Route Matching Order

When a user dials a number, the routing engine evaluates all enabled outbound routes in the following order:

  1. Sort by priority — Routes are sorted by priority ascending. Lower numbers are evaluated first.
  2. Pattern match — The dialed number is tested against each route's dial_pattern regex.
  3. First match wins — The first route whose pattern matches the dialed number is selected.
  4. No match — If no route matches, the call is rejected with a "no route to destination" error.

Recommended Priority Ranges

Priority Route Type Rationale
1 Emergency (911) Must always match first; never blocked by other patterns
10–20 N11 service codes Short codes like 411, 511 should match before general patterns
50 Long distance / domestic Standard outbound calls
100 Local dialing Default priority; evaluated after specific patterns
200+ International Broad prefix patterns; evaluated last to avoid false matches

Trunk Failover

The trunk_ids field is an ordered list. The first trunk in the list is the primary. If the primary trunk cannot complete the call (trunk down, capacity exceeded, or the carrier returns an error), the platform automatically tries the next trunk in the list, and so on until the call is placed or all trunks are exhausted.

Call matched to route │ ▼ Try trunk_ids[0] (Primary) ├─ Success → Call connected └─ Failure → Try trunk_ids[1] (Failover 1) ├─ Success → Call connected └─ Failure → Try trunk_ids[2] (Failover 2) ├─ Success → Call connected └─ Failure → All trunks exhausted → Call fails ("all circuits busy")

Failover happens automatically and is transparent to the caller (they hear ringing, not an error). The failover attempt adds a few seconds of delay per trunk, so keep the list to 2–3 trunks for optimal user experience.

Trunk Health Monitoring
The platform monitors trunk health via SIP OPTIONS pings. Known-down trunks are skipped immediately during failover, reducing delay. See SIP Trunks for monitoring configuration.

Configuration Examples

Emergency Route (Required)

{
  "name": "Emergency 911",
  "dial_pattern": "^911$",
  "strip_digits": 0,
  "cid_mode": "EXTENSION",
  "priority": 1,
  "enabled": true,
  "trunk_ids": ["aaaabbbb-cccc-dddd-eeee-ffffffffffff"]
}

US/Canada Domestic

{
  "name": "US Canada Domestic",
  "dial_pattern": "^1?[0-9]{10}$",
  "prepend_digits": "1",
  "strip_digits": 0,
  "cid_mode": "EXTENSION",
  "priority": 50,
  "enabled": true,
  "trunk_ids": [
    "aaaabbbb-cccc-dddd-eeee-ffffffffffff",
    "11112222-3333-4444-5555-666677778888"
  ]
}

International Dialing

{
  "name": "International",
  "dial_pattern": "^011[0-9]+$",
  "strip_digits": 3,
  "prepend_digits": "+",
  "cid_mode": "CUSTOM",
  "custom_cid": "+15551234567",
  "priority": 200,
  "enabled": true,
  "trunk_ids": ["33334444-5555-6666-7777-888899990000"]
}

Company-Wide Caller ID

{
  "name": "All Outbound - Company CID",
