Call Recordings

Manage, play back, and administer call recordings stored in object storage. Recordings are linked to CDRs and governed by per-extension recording policies.

Overview

The New Phone Platform can record calls and store the audio files in S3-compatible object storage (MinIO or AWS S3). Recordings are created based on the recording policy assigned to each extension and are linked to their corresponding Call Detail Record for easy retrieval.

Recording management includes secure playback through time-limited signed URLs, integrity verification via SHA-256 hashes, and configurable retention policies. All recordings are scoped to a tenant and protected by Row-Level Security.

Storage Backend
Recordings are stored in S3-compatible object storage (MinIO in self-hosted deployments, AWS S3 in cloud deployments). The platform handles all storage operations transparently — administrators interact with recordings through the API and web UI without needing direct access to the storage backend.

Recording Field Reference

The following table describes every field returned in a RecordingResponse object.

Field Type Description
id UUID Unique identifier for this recording
tenant_id UUID The tenant this recording belongs to (enforced by RLS)
cdr_id UUID | null Link to the associated Call Detail Record. Null in rare cases where the CDR has not yet been written at recording time.
call_id string The FreeSWITCH call identifier. Matches the call_id field in the corresponding CDR.
storage_path string | null The object key (path) within the S3/MinIO bucket where the recording file is stored
storage_bucket string | null The S3/MinIO bucket name containing the recording file
file_size_bytes integer Size of the recording file in bytes. Useful for estimating storage consumption and download times.
duration_seconds integer Length of the recording in seconds. May differ slightly from the CDR duration due to recording start/stop timing.
format string Audio file format: "wav" (uncompressed, higher quality) or "mp3" (compressed, smaller files)
sample_rate integer Audio sample rate in Hertz. Typically 8000 (narrowband/telephony) or 16000 (wideband/HD voice).
sha256_hash string | null SHA-256 hash of the recording file for integrity verification. Used to detect corruption or tampering.
recording_policy string The policy that was in effect when this recording was created: "never", "always", or "on_demand"
is_active boolean Whether the recording is active and accessible. Inactive recordings have been soft-deleted or archived.
created_at datetime Timestamp when the recording metadata was created in the database (ISO 8601, UTC)

Recording Policies

Each extension has a recording_policy that controls when calls involving that extension are recorded. The policy is evaluated at call setup time and determines whether the recording subsystem is activated for the call.

Policy Behavior Use Case
never Calls are never recorded, regardless of the other party's policy. No recording metadata or audio files are created. Executives, legal department, or any role where recording is prohibited by policy or regulation.
always Every call involving this extension is recorded automatically. Recording starts when the call is answered and stops when it ends. Call center agents, sales teams, support representatives. Required for compliance in regulated industries.
on_demand Recording does not start automatically. The user can initiate recording during a call using a feature code (typically *1) or a button in the softphone. Once started, recording continues until the call ends or the user stops it. General office staff who occasionally need to record a call for documentation purposes.

Policy Precedence

When two extensions with different policies are on a call together, the following precedence applies:

  • If either party has never, the call is not recorded
  • If either party has always (and neither has never), the call is recorded
  • If both parties have on_demand, recording only occurs if manually triggered
Changing Policies
Recording policies are configured on the extension settings page. Changes take effect for new calls only — calls already in progress when a policy changes continue under the original policy. See Extensions for configuration details.

Storage

Recordings are stored as individual audio files in S3-compatible object storage. The platform manages the full lifecycle: creation, storage, retrieval, and deletion.

Audio Format

Format Extension Typical Size Quality Notes
WAV .wav ~1 MB/min (8kHz) or ~2 MB/min (16kHz) Lossless Best quality. Preferred for compliance and legal holds. Larger file sizes.
MP3 .mp3 ~0.12 MB/min Lossy Significantly smaller files. Suitable for general business use where storage efficiency matters.

File Integrity

Each recording includes a sha256_hash field containing the SHA-256 digest of the audio file. This hash is computed when the recording is uploaded to object storage and can be used to:

  • Verify that a downloaded file has not been corrupted during transfer
  • Confirm that a recording has not been tampered with since creation
  • Meet chain-of-custody requirements for legal or regulatory proceedings

Storage Organization

Recordings are organized in object storage using a hierarchical key structure: 

{tenant_id}/{year}/{month}/{day}/{call_id}.{format}

This structure enables efficient lifecycle policies, bulk operations on date ranges, and logical isolation between tenants at the storage level.

Playback

Recordings are never served directly from object storage. Instead, the API generates time-limited signed URLs that grant temporary read access to the audio file. This approach ensures that recordings cannot be accessed without valid API authentication.

RecordingPlaybackResponse Schema

Field Type Description
url string A signed URL providing temporary read access to the recording file. Can be used directly in an audio player or for download.
expires_in_seconds integer Number of seconds until the signed URL expires. Default is 300 (5 minutes).

Playback Workflow

  1. Request a playback URL from the Recordings API by providing the recording ID
  2. The API verifies the user's permissions and tenant context
  3. A signed URL is generated with a 5-minute expiration window
  4. The client uses the signed URL to stream or download the audio file
  5. After expiration, the URL becomes invalid and a new one must be requested
URL Expiration
Signed playback URLs expire after 5 minutes by default. Do not cache or bookmark these URLs. If a URL expires during playback, the audio stream will be interrupted and the client must request a new URL. The web UI handles this automatically by requesting a fresh URL before each playback.

Filtering and Searching

The recording list endpoint accepts filter parameters to narrow results. All filters are optional and can be combined with AND logic.

RecordingFilter Parameters

Parameter Type Default Description
date_from datetime none Return recordings created on or after this timestamp (ISO 8601, UTC)
date_to datetime none Return recordings created on or before this timestamp (ISO 8601, UTC)
call_id string none Filter by the FreeSWITCH call identifier to find all recording segments for a specific call
cdr_id UUID none Filter by CDR identifier to find the exact recording for a specific call record
limit integer 50 Maximum number of records to return per request (1–1000)
offset integer 0 Number of records to skip before returning results (for pagination)

Pagination works identically to CDR pagination — see CDR Pagination for details on the offset-based model.

Linking Recordings to CDRs

Recordings and CDRs are linked through two fields, providing flexible lookup in both directions:

Lookup Direction Field Description
CDR → Recording CDR.has_recording Boolean on the CDR indicating whether a recording exists. If true, use the CDR's call_id or id to look up the recording.
Recording → CDR Recording.cdr_id Direct foreign key to the CDR. Use this to navigate from a recording back to its call details.
Shared key call_id Present on both CDR and Recording. Use when you need to find all artifacts (CDR, recordings, events) for a single call.

A single call may produce multiple recording segments if the call was transferred or if recording was started and stopped multiple times. In such cases, multiple recordings will share the same call_id but have different id values.

Legal Compliance

Call recording is subject to a complex patchwork of federal, state, and international laws. Administrators must understand and comply with the applicable regulations before enabling recording for any extension.

Consent Models

Model Requirement Jurisdictions (Examples)
One-Party Consent Only one participant in the call needs to know the call is being recorded. The person initiating the recording satisfies this requirement. US federal law (Wiretap Act), most US states, United Kingdom
Two-Party (All-Party) Consent All participants must be informed and consent to the recording before it begins. Typically satisfied by playing an announcement at the start of the call. California, Florida, Illinois, Pennsylvania, Washington, Canada (some provinces), Germany, many EU countries

Compliance Recommendations

  • Always announce recording — Play a recording announcement at the start of recorded calls. This satisfies both one-party and two-party consent requirements and is considered best practice regardless of jurisdiction.
  • Use IVR announcements — Configure your IVR menus to include a recording disclosure before routing to agents. See IVR Menus.
  • Document your policy — Maintain a written call recording policy that specifies which calls are recorded, why, how recordings are stored, who has access, and how long they are retained.
  • Restrict access — Use RBAC to limit recording playback to authorized personnel only. See Roles & Permissions.
  • Consult legal counsel — Laws vary by jurisdiction and change frequently. Obtain legal advice specific to your operating locations before finalizing your recording policy.
Legal Disclaimer
This documentation provides general guidance only and does not constitute legal advice. Recording laws vary significantly by jurisdiction and are subject to change. Consult with qualified legal counsel to ensure your recording practices comply with all applicable laws.

Storage Tiering Coming Soon

A future release will introduce automatic storage tiering to optimize costs for long-term recording retention. Recordings will automatically transition through storage classes based on their age:

  • Hot tier (0–30 days) — Standard storage with immediate access. Used for recent recordings that are frequently played back.
  • Warm tier (30–90 days) — Infrequent access storage with slightly higher retrieval latency. Reduced storage cost.
  • Cold tier (90+ days) — Archive storage with multi-hour retrieval times. Lowest storage cost. Suitable for compliance retention where immediate access is not required.

Transition rules will be configurable per tenant by MSP administrators. Until this feature is available, all recordings remain in standard storage regardless of age.

Retention and Deletion

Recording retention is managed separately from CDR retention, since recordings consume significantly more storage space. The default retention policy deletes recording audio files after 90 days while preserving the recording metadata record (which references the CDR).

Default Retention Settings

Component Retention Notes
Audio file (object storage) 90 days The actual audio file is deleted from S3/MinIO. Playback is no longer possible.
Recording metadata (database) Matches CDR retention The database record persists to maintain CDR linkage and audit history, even after the audio file is deleted.

Deletion Behavior

  • When a recording audio file is deleted, the is_active field is set to false
  • The storage_path and storage_bucket fields are cleared
  • Attempts to generate a playback URL for an inactive recording return an error
  • Metadata (duration, format, hash, timestamps) remains available for historical reference
  • Deletion is permanent and cannot be undone — there is no recycle bin for recordings

Legal Hold

Recordings can be placed on legal hold to prevent automatic deletion. When a recording is under legal hold, the retention policy is suspended and the audio file is preserved indefinitely until the hold is released by an administrator. Use legal holds when recordings may be needed for litigation, regulatory investigation, or dispute resolution.

Storage Cost Impact
Recording audio files are typically the largest data category in the platform. A single tenant with 50 extensions making 100 calls per day at 3 minutes average duration generates approximately 300 MB/day in WAV format or 36 MB/day in MP3 format. Plan storage capacity and budget accordingly, especially when extending retention periods.
