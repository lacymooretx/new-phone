Time Conditions

Route calls differently based on time of day, day of week, date ranges, and holiday calendars. Define business hours, after-hours, and emergency overrides.

Overview

Time conditions control how incoming calls are routed based on the current date and time. Each time condition defines a set of rules that determine whether the current moment falls within "business hours" (match) or "after hours" (no-match), and routes calls to the appropriate destination accordingly.

Common uses include routing calls to a ring group during working hours and to voicemail or an after-hours IVR outside of business hours. Time conditions can also be linked to holiday calendars so that recognized holidays automatically override normal business-hour rules.

Inbound Route Integration
Time conditions are typically used as destinations in inbound routes. When a call arrives, the inbound route sends it to a time condition, which evaluates the current time and forwards the call to either the match or no-match destination.

Creating Time Conditions

To create a time condition, navigate to System > Time Conditions and click Add Time Condition. Provide a descriptive name, configure one or more rules, and set the match and no-match destinations.

A time condition with no rules will always evaluate to no-match. At minimum, you should define a day-of-week rule and a time-of-day rule to represent standard business hours.

Field Reference

TimeConditionCreate Fields

Field Type Required Description
name string Yes Descriptive name (1–255 characters). Example: "Main Office Hours"
description string No Optional longer description of the time condition's purpose
timezone string No IANA timezone identifier. Default: "America/New_York"
rules list[TimeConditionRule] No List of rules that must all match for the condition to evaluate as "match"
match_destination_type string Yes Destination type when rules match (e.g., ring_group, ivr, queue, extension, voicemail)
match_destination_id UUID No ID of the match destination resource
nomatch_destination_type string Yes Destination type when rules do not match (after-hours routing)
nomatch_destination_id UUID No ID of the no-match destination resource
holiday_calendar_id UUID No Linked holiday calendar. Holidays override normal rules and force no-match
enabled boolean No Whether the time condition is active. Default: true

TimeConditionRule Fields

Field Type Required Description
type string Yes One of: day_of_week, time_of_day, specific_date, date_range
days list[int] Conditional Required for day_of_week. Days of week: 1=Monday through 7=Sunday
start_time string Conditional Required for time_of_day and date_range. Format: HH:MM
end_time string Conditional Required for time_of_day and date_range. Format: HH:MM
start_date string Conditional Required for specific_date and date_range. Format: YYYY-MM-DD
end_date string Conditional Required for date_range. Format: YYYY-MM-DD
invert boolean No Negate the rule result. Default: false
label string No Optional description for this rule (e.g., "Weekdays only")

TimeConditionUpdate Additional Fields

The update schema includes all fields from TimeConditionCreate (all optional), plus:

Field Type Description
manual_override string Force a specific result: "match" or "nomatch". Set to null to clear

Response Fields

The response includes all create fields plus the following system-managed fields:

Field Type Description
id UUID Unique identifier for the time condition
tenant_id UUID Owning tenant (set automatically from auth context)
manual_override string | null Current manual override state, if any
is_active boolean Current evaluation result (match = true, no-match = false)
created_at datetime Timestamp when the time condition was created
updated_at datetime Timestamp of the last modification

Rule Types Explained

day_of_week

Matches when the current day falls within the specified list of days. Days are numbered 1 (Monday) through 7 (Sunday), following the ISO 8601 standard. 

{
  "type": "day_of_week",
  "days": [1, 2, 3, 4, 5],
  "label": "Weekdays (Mon-Fri)"
}

time_of_day

Matches when the current time (in the configured timezone) falls between start_time and end_time. Times use 24-hour format. 

{
  "type": "time_of_day",
  "start_time": "08:00",
  "end_time": "17:00",
  "label": "Business hours (8 AM - 5 PM)"
}

specific_date

Matches when the current date is exactly the specified date. Useful for one-off events like office closures or special schedules. Uses start_date only. 

{
  "type": "specific_date",
  "start_date": "2026-12-24",
  "label": "Christmas Eve closure"
}

date_range

Matches when the current date falls within a range (inclusive) and optionally the current time falls within a time window. Useful for seasonal schedules or temporary changes. 

{
  "type": "date_range",
  "start_date": "2026-06-01",
  "end_date": "2026-08-31",
  "start_time": "08:00",
  "end_time": "16:00",
  "label": "Summer hours (Jun-Aug, 8 AM - 4 PM)"
}

Rule Evaluation Logic

Time conditions use AND logic across all rules. Every rule in the list must evaluate to true for the overall condition to produce a "match" result. If any single rule fails, the condition evaluates to "no-match."

This AND behavior makes it natural to combine a day-of-week rule with a time-of-day rule: both must be true simultaneously, meaning the call must arrive on the right day and at the right time.

Inverted Rules
When a rule has "invert": true, its result is negated before being combined with the other rules. For example, a day_of_week rule for Saturday and Sunday with invert: true effectively means "not Saturday and not Sunday" — which matches on weekdays only.

Match vs No-Match Destinations

Every time condition requires two destinations:

  • Match destination — Where calls are routed when all rules evaluate to true (business hours). Typically a ring group, queue, or IVR.
  • No-match destination — Where calls are routed when any rule fails (after hours). Typically a voicemail box, after-hours IVR, or announcement.

Destination types include: extension, ring_group, queue, ivr, voicemail, conference, announcement, and time_condition (for chaining).

Chaining Time Conditions
You can set a time condition's destination to another time condition. This allows complex routing logic, such as checking business hours first and then checking for a lunch-hour schedule within those hours.

Holiday Calendar Integration

When a holiday calendar is linked to a time condition, the system checks the calendar before evaluating the time rules. If the current date and time fall within a holiday entry, the time condition immediately evaluates to no-match, bypassing all rules.

This means calls on holidays are automatically routed to the after-hours destination without requiring manual intervention. See Holiday Calendars for details on creating and managing calendars.

Manual Override

The manual_override field allows administrators to force a time condition to a specific state, regardless of the current rules. This is useful for emergency situations, such as office closures due to weather or unexpected events.

Override Value Behavior
"match" Force business-hours routing regardless of time/day. Calls go to match destination.
"nomatch" Force after-hours routing regardless of time/day. Calls go to no-match destination.
null Normal operation. Rules are evaluated as configured.
Remember to Clear Overrides
Manual overrides persist until explicitly cleared. After an emergency passes, update the time condition and set manual_override to null to restore normal rule-based evaluation.

Timezone Handling

Each time condition has its own timezone setting. All rule evaluations use the configured timezone to determine the current local time. The timezone must be a valid IANA timezone identifier.

Common timezone values:

  • America/New_York — Eastern Time
  • America/Chicago — Central Time
  • America/Denver — Mountain Time
  • America/Los_Angeles — Pacific Time
  • America/Phoenix — Arizona (no DST)
  • Pacific/Honolulu — Hawaii
  • Europe/London — UK

Daylight Saving Time transitions are handled automatically. The system uses the IANA timezone database, which is updated regularly. When DST transitions occur, rule evaluation shifts accordingly without any manual changes.

Common Examples

Standard Business Hours (Monday–Friday, 8 AM–5 PM)

{
  "name": "Main Office Hours",
  "timezone": "America/New_York",
  "rules": [
    {
      "type": "day_of_week",
      "days": [1, 2, 3, 4, 5],
      "label": "Weekdays"
    },
    {
      "type": "time_of_day",
      "start_time": "08:00",
      "end_time": "17:00",
      "label": "8 AM to 5 PM"
    }
  ],
  "match_destination_type": "ring_group",
  "match_destination_id": "uuid-of-main-ring-group",
  "nomatch_destination_type": "voicemail",
  "nomatch_destination_id": "uuid-of-main-voicemail",
  "holiday_calendar_id": "uuid-of-us-holidays"
}

Calls during weekdays between 8 AM and 5 PM Eastern go to the main ring group. All other times (evenings, weekends, and US holidays) route to voicemail.

Weekend-Only Routing

{
  "name": "Weekend Support",
  "timezone": "America/Chicago",
  "rules": [
    {
      "type": "day_of_week",
      "days": [6, 7],
      "label": "Saturday and Sunday"
    },
    {
      "type": "time_of_day",
      "start_time": "10:00",
      "end_time": "14:00",
      "label": "10 AM to 2 PM"
    }
  ],
  "match_destination_type": "queue",
  "match_destination_id": "uuid-of-weekend-queue",
  "nomatch_destination_type": "announcement",
  "nomatch_destination_id": "uuid-of-closed-announcement"
}

Holiday Routing with Calendar

Link a holiday calendar to your business-hours time condition. On recognized holidays, calls skip the normal rules and route directly to the no-match (after-hours) destination. No per-holiday configuration is needed in the time condition itself. 

{
  "name": "Reception Hours",
  "timezone": "America/New_York",
  "rules": [
    { "type": "day_of_week", "days": [1, 2, 3, 4, 5] },
    { "type": "time_of_day", "start_time": "09:00", "end_time": "17:30" }
  ],
  "match_destination_type": "ivr",
  "match_destination_id": "uuid-of-main-ivr",
  "nomatch_destination_type": "ivr",
  "nomatch_destination_id": "uuid-of-afterhours-ivr",
  "holiday_calendar_id": "uuid-of-company-holidays"
}
