Tenant Management MSP Only

Create and manage isolated customer environments in the multi-tenant PBX platform.

Overview

A tenant represents an isolated customer environment within the Aspendora Connect. Each tenant has its own extensions, call routing, users, recordings, and configuration. Tenants are the foundation of the platform's multi-tenant architecture, allowing a single deployment to serve many independent organizations without any data leakage between them.

Managed Service Providers (MSPs) use tenants to onboard customer organizations. Each tenant receives a unique identifier, an optional custom domain, and a dedicated SIP domain. All telephony resources, user accounts, and call data are scoped to the tenant boundary.

MSP-Only Feature
Tenant management is restricted to MSP-level roles. Tenant administrators and users cannot create, modify, or view other tenants. They operate exclusively within their own tenant boundary.

Creating a Tenant

To create a new tenant, submit a POST request to /api/v1/tenants with a JSON body matching the TenantCreate schema. At minimum, you must provide a name and slug.

  1. Navigate to MSP > Tenants in the admin interface.
  2. Click Create Tenant.
  3. Fill in the required fields: name and slug.
  4. Optionally configure domain, sip_domain, music-on-hold, and notes.
  5. Click Save. The tenant is created and immediately active.

TenantCreate Field Reference

FieldTypeRequiredConstraintsDescription

slug string Yes 1–63 chars, pattern: ^[a-z0-9][a-z0-9-]*$ URL-safe identifier used in API paths and subdomain routing. Must start with a letter or digit, followed by lowercase letters, digits, or hyphens.
domain string No — Custom domain for the tenant (e.g., pbx.acmecorp.com).
sip_domain string No — SIP domain for phone registrations and routing. If omitted, defaults to <slug>.sip.newphone.io.

notes string No — Internal notes visible only to MSP staff. Useful for account notes, contract details, or special instructions.

Editing a Tenant

To update an existing tenant, send a PATCH request to /api/v1/tenants/{tenant_id} with a JSON body matching the TenantUpdate schema. All fields are optional; only the fields you include will be modified.

The TenantUpdate schema accepts all fields from TenantCreate (each optional) plus an additional is_active boolean field for enabling or disabling the tenant.

Changing the Slug
Updating a tenant's slug will affect API paths and any subdomain-based routing. Coordinate slug changes with DNS and SIP configuration to avoid service disruption.

Deactivating vs Deleting Tenants

The platform uses soft deletion via the is_active flag rather than permanent deletion. This preserves call records, recordings, and audit trails for compliance purposes.

ActionEffectReversible
Deactivate (is_active: false) All phone registrations are dropped. Users cannot log in. Inbound calls receive a not-in-service treatment. All data is preserved. Yes — set is_active: true to restore.
Hard delete Not available through the API. Data retention policies and compliance requirements prohibit permanent deletion without a formal data purge process. N/A
Best Practice
When offboarding a customer, deactivate the tenant first and allow a retention period (typically 90 days) before initiating any data purge procedures.

Row-Level Security

Tenant isolation is enforced at the database level using PostgreSQL Row-Level Security (RLS). Every table that contains tenant-scoped data includes a tenant_id column, and RLS policies ensure that queries can only access rows belonging to the current tenant.

How It Works

  • The API application connects to PostgreSQL as the new_phone_app role, which has RLS enforced.
  • At the start of each database transaction, the API sets SET LOCAL app.current_tenant = '<tenant_id>'.
  • RLS policies on every tenant-scoped table filter rows where tenant_id = current_setting('app.current_tenant')::uuid.
  • The SET LOCAL scope ensures the setting is automatically cleared when the transaction ends, preventing any cross-tenant leakage through connection pooling.

Migrations and administrative operations use the new_phone_admin role, which bypasses RLS. This two-role pattern ensures that application code can never accidentally access another tenant's data, even if there is a bug in the API layer.

SIP Domain Configuration

Each tenant can have a dedicated SIP domain used for phone registration and call routing. The SIP domain determines how phones authenticate and where calls are directed.

Default Behavior

If sip_domain is not explicitly set during tenant creation, the platform generates one automatically using the pattern <slug>.sip.newphone.io. For example, a tenant with slug acme-corp would receive the SIP domain acme-corp.sip.newphone.io.

Custom SIP Domains

To use a custom SIP domain, set the sip_domain field and configure the appropriate DNS records (SRV and A/AAAA) to point to the platform's SIP proxy. All SIP connections require TLS; unencrypted SIP is not supported.

Examples

Create a Tenant

POST /api/v1/tenants
Content-Type: application/json
Authorization: Bearer <msp_token>

{
  "name": "Acme Corporation",
  "slug": "acme-corp",
  "domain": "pbx.acmecorp.com",
  "sip_domain": "acme-corp.sip.newphone.io",
  "notes": "Contract #4521 - 50 seats"
}

Update a Tenant

PATCH /api/v1/tenants/<tenant_id>
Content-Type: application/json
Authorization: Bearer <msp_token>

{
  "name": "Acme Corp (Renamed)",
  "notes": "Upgraded to 100 seats - Jan 2026"
}

Deactivate a Tenant

PATCH /api/v1/tenants/<tenant_id>
Content-Type: application/json
Authorization: Bearer <msp_token>

{
  "is_active": false
}
