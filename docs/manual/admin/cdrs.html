Call Detail Records

Every call that passes through the platform generates a Call Detail Record (CDR). Use CDRs to analyze call volume, troubleshoot issues, track agent performance, and produce billing reports.

Overview

A Call Detail Record is created automatically for every call that enters, leaves, or occurs within the platform. CDRs capture the full lifecycle of a call from the moment it begins ringing until the final hangup. They are immutable once written — no user or administrator can modify a CDR after it has been recorded.

CDRs serve as the authoritative source for call history, billing calculations, compliance auditing, and performance analysis. Each CDR is scoped to a single tenant and respects Row-Level Security, meaning users can only see records belonging to their own tenant.

Automatic Creation
CDRs are generated by the platform automatically at the conclusion of every call. There is no manual step required. Even calls that fail to connect (busy, no answer, network failure) produce a CDR so that every attempt is accounted for.

CDR Field Reference

The following table describes every field returned in a CDRResponse object.

Field Type Description
id UUID Unique identifier for this CDR record
tenant_id UUID The tenant this call belongs to (enforced by RLS)
call_id string Unique call identifier assigned by FreeSWITCH at call creation. Used to correlate CDRs with recordings, events, and logs.
direction enum Call direction: INBOUND, OUTBOUND, or INTERNAL
caller_number string The originating phone number or extension number of the caller
caller_name string Caller ID name as presented by the originating party or trunk
called_number string The destination number or extension that was dialed
extension_id UUID | null The extension involved in the call, if applicable. Null for trunk-to-trunk or system-generated calls.
did_id UUID | null The DID (phone number) used for inbound calls. Null for outbound and internal calls.
trunk_id UUID | null The SIP trunk used for the call. Null for internal extension-to-extension calls.
ring_group_id UUID | null The ring group that handled the call, if any. Null when the call did not route through a ring group.
disposition enum Final outcome of the call: ANSWERED, NO_ANSWER, BUSY, FAILED, VOICEMAIL, or CANCELLED
hangup_cause string | null The SIP/Q.850 hangup cause code. Provides detailed technical reason for call termination. See SIP Hangup Causes.
duration_seconds integer Total elapsed time of the call in seconds, from start to end (includes ringing time)
billable_seconds integer Billable duration after the grace period. Zero for unanswered calls. See Duration Fields.
ring_seconds integer Time in seconds from the call start until the call was answered (or ended if not answered)
start_time datetime Timestamp when the call was initiated (ISO 8601, UTC)
answer_time datetime | null Timestamp when the call was answered. Null if the call was never answered.
end_time datetime Timestamp when the call ended (ISO 8601, UTC)
has_recording boolean Whether a call recording exists for this CDR. When true, use the Recordings API to retrieve it.
created_at datetime Timestamp when the CDR was written to the database

Direction Types

Every CDR is classified by the direction of the call. The direction is determined automatically based on how the call entered or was initiated within the platform.

Direction Description Example
INBOUND A call that arrived from the PSTN via a SIP trunk and was routed to an internal destination (extension, queue, ring group, IVR, or voicemail). An external customer calls your main office number.
OUTBOUND A call that originated from an internal extension and was sent out to the PSTN via a SIP trunk. An employee dials a customer's cell phone from their desk phone.
INTERNAL A call between two extensions within the same tenant. These calls never touch a SIP trunk and incur no carrier charges. Extension 101 calls extension 205 for an internal discussion.
Trunk and DID Context
For INBOUND calls, the trunk_id and did_id fields identify which SIP trunk and DID number received the call. For OUTBOUND calls, trunk_id identifies the trunk used for egress. For INTERNAL calls, both fields are null.

Disposition Types

The disposition indicates the final outcome of the call. It is set when the call terminates and reflects what happened from the caller's perspective.

Disposition Description Typical Scenario
ANSWERED The call was picked up by a human or system and a conversation (or connection) occurred. The answer_time field will be populated, and billable_seconds will be greater than zero. A support agent answers a customer call.
NO_ANSWER The call rang but nobody picked up before the ring timeout expired. The call was not forwarded to voicemail. An extension rings for 30 seconds with no pickup and no voicemail configured.
BUSY The destination returned a busy signal, typically because all lines were occupied or the device rejected the call with a 486 response. A single-line device is already on a call when another call arrives.
FAILED The call could not be completed due to a network or system error. This includes routing failures, trunk capacity issues, and SIP protocol errors. An outbound call fails because the SIP trunk is unreachable.
VOICEMAIL The call was forwarded to voicemail and the caller left a message (or the voicemail system answered). The answer_time reflects when the voicemail system picked up. No one answers and the call forwards to voicemail after the ring timeout.
CANCELLED The caller hung up before the call was answered or before it finished ringing. The call was abandoned by the originator. A customer in a queue hangs up after waiting 45 seconds.

Duration Fields

CDRs track three separate duration measurements. Understanding the difference is important for billing accuracy and performance analysis.

Field Start Point End Point Use Case
duration_seconds start_time end_time Total elapsed time, including ringing. Useful for understanding total call lifecycle and agent availability impact.
billable_seconds answer_time + grace period end_time Duration used for billing after the carrier grace period. Zero for unanswered calls. Used in cost calculations and trunk usage reports.
ring_seconds start_time answer_time (or end_time if unanswered) How long the phone rang before being answered. Key metric for queue SLA tracking and response time analysis.

The grace period is a configurable window (typically 6 seconds) at the beginning of an answered call during which no billing accrues. This accounts for the brief connection time before a real conversation begins. If a call is answered and hangs up within the grace period, billable_seconds will be zero even though the call was technically answered.

Duration Calculation
duration_seconds always equals ring_seconds + talk time. For unanswered calls, duration_seconds equals ring_seconds since there was no talk time. The billable_seconds value is always less than or equal to the talk time portion.

Filtering and Searching

The CDR list endpoint accepts a set of filter parameters to narrow results. All filters are optional and can be combined. When multiple filters are provided, they are applied with AND logic — only records matching all specified criteria are returned.

CDRFilter Parameters

Parameter Type Default Description
date_from datetime none Return only CDRs with start_time on or after this timestamp (ISO 8601, UTC)
date_to datetime none Return only CDRs with start_time on or before this timestamp (ISO 8601, UTC)
extension_id UUID none Filter by a specific extension involved in the call
direction CDRDirection none Filter by call direction: INBOUND, OUTBOUND, or INTERNAL
disposition CDRDisposition none Filter by call outcome: ANSWERED, NO_ANSWER, BUSY, FAILED, VOICEMAIL, or CANCELLED
limit integer 50 Maximum number of records to return per request (1–1000)
offset integer 0 Number of records to skip before returning results (for pagination)

Common Filter Combinations

  • All missed calls today — Set date_from to today's midnight, disposition to NO_ANSWER
  • Outbound calls for a specific extension — Set extension_id and direction to OUTBOUND
  • Failed calls this week — Set date_from to the start of the week, disposition to FAILED
  • All inbound calls to voicemail — Set direction to INBOUND and disposition to VOICEMAIL

Pagination

CDR results are paginated using limit and offset parameters. This is an offset-based pagination model suitable for browsing through call history.

  • First page: limit=50&offset=0 (default)
  • Second page: limit=50&offset=50
  • Third page: limit=50&offset=100

The maximum value for limit is 1000 records per request. For large datasets, consider using date range filters to narrow the result set before paginating, which improves query performance significantly.

Offset Pagination Caveat
Offset-based pagination can produce inconsistent results if new CDRs are created between page requests (since new records shift offsets). For real-time monitoring, use the dashboard live view instead. For bulk exports, use the export endpoint with fixed date ranges.

Linking to Recordings

The has_recording boolean field on each CDR indicates whether a call recording exists for that call. When this field is true, you can use the call_id from the CDR to look up the corresponding recording via the Recordings API.

  • Filter recordings by cdr_id to find the exact recording for a specific CDR
  • Filter recordings by call_id to find all recordings associated with a call (useful if a call was transferred and produced multiple recording segments)
  • The recording's duration_seconds may differ slightly from the CDR's billable_seconds due to recording start/stop timing

See the Recordings page for details on playback, storage, and signed URL generation.

SIP Hangup Causes

The hangup_cause field contains the technical reason why a call ended. These codes follow the ITU-T Q.850 standard and are mapped from SIP response codes by FreeSWITCH. This field is invaluable for troubleshooting failed or abnormally terminated calls.

Common Hangup Causes

Cause Code Name SIP Code Meaning
16 NORMAL_CLEARING 200 BYE Normal call hangup. One party ended the call gracefully. This is the expected cause for most completed calls.
17 USER_BUSY 486 The destination returned a busy indication. The called device was occupied or rejected the call.
18 NO_USER_RESPONSE 408 The destination did not respond within the timeout period. The device may be offline or unreachable.
19 NO_ANSWER 480 The destination rang but was not answered before the ring timeout expired.
21 CALL_REJECTED 403 The call was explicitly rejected by the destination or an intermediary. May indicate a permissions issue or call barring.
27 DESTINATION_OUT_OF_ORDER 502 The destination endpoint is malfunctioning or unreachable. Common when a SIP trunk is down.
28 INVALID_NUMBER_FORMAT 484 The dialed number has an invalid format. Check outbound route dial patterns and caller-entered digits.
34 NO_CIRCUIT_AVAILABLE 503 No trunk channels are available. The trunk has reached its maximum concurrent call limit.
38 NETWORK_OUT_OF_ORDER 503 A network-level failure prevented the call from completing. Check connectivity between the platform and the SIP provider.
127 INTERWORKING 500 A generic interworking error occurred. Often a catch-all when the specific cause cannot be determined.

When investigating call failures, start with the hangup_cause to identify the category of problem (network, capacity, destination, or configuration), then cross-reference with the FreeSWITCH logs for the corresponding call_id for deeper diagnostics.

Export and Reporting

CDR data can be exported for use in external reporting tools, billing systems, or compliance archives. The platform supports the following export workflows:

CSV Export

  • Export filtered CDR results to CSV format from the web UI
  • All active filters are applied to the export — export exactly what you see
  • CSV files include all CDR fields with headers matching the API field names
  • Timestamps are exported in ISO 8601 format (UTC)
  • Large exports are processed asynchronously and delivered as a download link

API-Based Export

  • Use the CDR list endpoint with appropriate filters and a high limit value
  • For datasets larger than 1000 records, paginate through the full result set
  • Combine with date range filters for efficient batch retrieval
  • JSON responses can be transformed into any format by your reporting pipeline

Reporting Tips

  • Daily call volume — Group CDRs by start_time date and count by direction
  • Average handle time — Calculate the mean of duration_seconds for ANSWERED calls per extension
  • Answer rate — Divide ANSWERED count by total calls for a given period
  • Peak hours — Group by hour of day to identify call volume patterns
  • Trunk utilization — Count concurrent calls per trunk_id to assess capacity needs

Retention Policy

CDR data is retained according to the tenant's configured retention policy. The default retention settings are as follows:

Tier Retention Period Storage
Active 90 days Primary database (fast queries, full filtering)
Archive 1 year Compressed archive (slower queries, date-range access only)
Cold Storage 7 years Object storage (S3/MinIO). Requires manual retrieval request.

MSP administrators can adjust retention periods per tenant to meet specific regulatory or contractual requirements. Extending retention beyond the default may affect storage costs.

Compliance Note
Some industries (healthcare, finance, government) require call records to be retained for specific periods. Ensure your retention policy meets applicable regulations such as HIPAA, PCI-DSS, or state-level record retention laws before reducing the default retention period.
