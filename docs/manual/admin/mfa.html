Multi-Factor Authentication (MFA)

Add a second layer of security to user accounts with time-based one-time passwords.

What is MFA?

Multi-Factor Authentication (MFA) adds a second verification step to the login process. After entering their email and password, users must also provide a one-time code from a separate device. This ensures that even if a password is compromised, an attacker cannot access the account without the second factor.

MFA significantly reduces the risk of account takeover from phishing, credential stuffing, and password reuse attacks. It is strongly recommended for all users and especially critical for administrator accounts.

Best Practice
MSP and tenant administrators should enable MFA on their own accounts before requiring it for other users. This ensures familiarity with the setup process and provides a fallback in case users need assistance.

Supported Methods

TOTP (Authenticator Apps) — Available Now

Time-Based One-Time Password (TOTP) is the currently implemented MFA method. It works with any standard authenticator app that supports the TOTP algorithm (RFC 6238).

Compatible authenticator apps include:

  • Google Authenticator (iOS, Android)
  • Microsoft Authenticator (iOS, Android)
  • Authy (iOS, Android, Desktop)
  • 1Password, Bitwarden, and other password managers with TOTP support

TOTP generates a new 6-digit code every 30 seconds. The code is derived from a shared secret between the server and the authenticator app, combined with the current time.

WebAuthn (Security Keys) Coming Soon

WebAuthn support will allow users to authenticate with hardware security keys (YubiKey, Titan Key) or platform authenticators (Touch ID, Windows Hello, Face ID). This provides phishing-resistant authentication that does not rely on shared secrets.

SMS Verification Coming Soon

SMS-based verification will be available as a fallback option. While less secure than TOTP or WebAuthn, it provides a familiar method for users who may not be comfortable with authenticator apps.

SMS Security Note
SMS-based MFA is vulnerable to SIM-swapping attacks and should only be used as a fallback. TOTP or WebAuthn are strongly preferred for all accounts.

Setup Flow

MFA setup is a two-step process: the server generates a secret and provides it to the user (via QR code or manual entry), and then the user verifies they have correctly configured their authenticator by submitting a valid code.

Setup Sequence

User → Request MFA Setup → API ← MFASetupResponse (secret, qr_code, provisioning_uri) User → Scan QR code with authenticator app → Authenticator generates TOTP codes User → MFAVerifyRequest (code) → API ← Success: MFA is now enabled ← Failure: Invalid code, try again

MFASetupResponse Schema

Returned when a user initiates MFA setup. Contains everything needed to configure an authenticator app.

Field Type Description
secret string Base32-encoded shared secret for manual entry into authenticator apps
qr_code string Base64-encoded PNG image of the QR code for scanning
provisioning_uri string otpauth:// URI that encodes the secret, issuer, and account name. Can be used directly by authenticator apps.

MFAVerifyRequest Schema

Submit a TOTP code to confirm the authenticator is correctly configured.

Field Type Required Description
code string Yes 6-digit TOTP code from the authenticator app

Step-by-Step Setup Procedure

Follow these steps to enable MFA on a user account.

  1. Navigate to Security Settings
    Open the user profile or, as an admin, navigate to the user's account in the Users management page. Locate the MFA section.
  2. Initiate MFA Setup
    Click the Enable MFA button. The API generates a new TOTP secret and returns the MFASetupResponse containing the QR code, secret, and provisioning URI.
  3. Open Your Authenticator App
    Launch Google Authenticator, Microsoft Authenticator, Authy, or any compatible TOTP app on your mobile device.
  4. Scan the QR Code
    Use the authenticator app's "Scan QR Code" or "Add Account" feature to scan the displayed QR code. If scanning is not possible, manually enter the Base32 secret shown on screen.
  5. Verify the Setup
    The authenticator app will immediately begin generating 6-digit codes that refresh every 30 seconds. Enter the current code into the verification field and click Verify.
  6. Save Recovery Information
    Store the Base32 secret in a secure location (such as a password manager) as a recovery backup. If you lose access to your authenticator device, this secret can be used to re-register with a new app.
  7. Confirmation
    If the code is valid, MFA is now active on the account. All future logins will require both the password and a TOTP code. The user interface will update to show MFA as enabled.
Setup Complete
Once MFA is verified and active, the user will be prompted for a TOTP code at every login. The MFA challenge is inserted between password verification and token issuance, as described in the Authentication & Login page.

Disabling MFA

MFA can be disabled under specific circumstances. This is an administrative action that should be performed carefully.

Self-Disable (Users)

Users can disable their own MFA from their profile security settings. They must first authenticate with their current TOTP code to confirm the action. This prevents an attacker who has obtained only the password from disabling MFA.

Admin Disable

Tenant administrators and MSP administrators can disable MFA for any user within their scope. This is typically used when a user has lost access to their authenticator device and cannot log in. The action is recorded in the Audit Log.

Security Consideration
After an admin disables MFA for a user, the user should re-enable MFA as soon as possible. Consider also resetting the user's password to ensure full credential rotation if the disable was triggered by a suspected compromise.

Recovery Options

If a user loses access to their authenticator device, the following recovery methods are available:

Re-register with Saved Secret

If the user saved the Base32 secret during initial setup, they can add it to a new authenticator app and resume generating codes immediately. No changes are needed on the server side.

Admin-Assisted Recovery

An administrator can disable MFA on the user's account, allowing the user to log in with only their password. After logging in, the user should immediately set up MFA again with a new secret.

Backup Codes Coming Soon

A future release will include one-time backup codes generated during MFA setup. Each code can be used once as an alternative to the TOTP code, providing a self-service recovery path without administrator involvement.

Troubleshooting

Problem Likely Cause Solution
Code is always rejected Device clock is out of sync Ensure the device's time is set to automatic. TOTP depends on accurate time.
QR code does not scan Camera issue or low screen brightness Increase screen brightness or manually enter the Base32 secret.
Lost authenticator device Device lost, stolen, or reset Use the saved Base32 secret on a new device, or request admin-assisted recovery.
MFA prompt not appearing at login MFA may not be enabled on the account Check the user's security settings to confirm MFA is active.
Code works once then fails TOTP codes are time-limited (30s window) Wait for a fresh code to appear before entering it. Do not reuse a code.
