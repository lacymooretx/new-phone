SIP Trunk Configuration

Connect the PBX to PSTN providers using SIP trunks with mandatory TLS signaling and SRTP media encryption.

Overview & Security

SIP trunks are the connections between the New Phone Platform and external telephony providers (PSTN gateways). Each trunk defines how calls are routed to and from the public telephone network, including authentication credentials, codec preferences, and capacity limits.

By design, all SIP signaling is TLS-mandatory and all media is SRTP-mandatory. Unencrypted SIP (UDP/TCP) and unencrypted RTP are not permitted. This ensures that call setup metadata and voice media are protected in transit at all times.

Encryption Is Non-Negotiable
The platform enforces SIP over TLS and SRTP for all trunk connections. The transport field defaults to TLS and should not be changed to TCP or UDP unless you are operating in a lab environment with full understanding of the security implications. Production deployments must use TLS.
PBX ──[SIP TLS]──> Provider (host:port) │ │ │ Auth: IP_ACL │ Verify source IP │ Auth: USERNAME │ SIP REGISTER with credentials │ Auth: MUTUAL_TLS │ Client certificate exchange │ │ └──[SRTP]──media──> │

The diagram above illustrates the three authentication modes available when connecting to a provider. Regardless of which mode is chosen, both the signaling path (SIP over TLS) and the media path (SRTP) are always encrypted.

Authentication Types

Each trunk must use one of three authentication methods. The method you choose depends on what your provider supports and your security requirements.

IP ACL (Access Control List)

IP-based authentication relies on a list of allowed IP addresses or CIDR ranges. The provider trusts traffic from your PBX based solely on its source IP. No username or password is exchanged during call setup.

  • Set auth_type to IP_ACL
  • Populate the ip_acl field with allowed IPs or CIDR notation (e.g., 203.0.113.0/24)
  • The username and password fields are not used
  • Best for static-IP deployments where both sides have predictable network addresses

Username & Password

Digest authentication using SIP REGISTER. The PBX sends credentials to the provider to authenticate. This is the most common method offered by SIP trunk providers.

  • Set auth_type to USERNAME_PASSWORD
  • Provide username and password
  • The password is encrypted at rest in the database and is never returned by the API
  • The PBX will register with the provider at regular intervals to maintain the trunk
Password Security
Trunk passwords are encrypted in the database using AES-256. The API will never include the password or encrypted_password fields in any response. To change a password, submit a new value in an update request.

Mutual TLS

Both the PBX and the provider present TLS client certificates during the handshake. This provides the strongest authentication because both sides cryptographically prove their identity.

  • Set auth_type to MUTUAL_TLS
  • Requires client certificate configuration on both the PBX and the provider
  • No username, password, or ip_acl fields are needed
  • Certificate management (issuance, renewal, revocation) is handled outside the trunk configuration

Creating a Trunk

Follow these steps to add a new SIP trunk to your tenant.

  1. Navigate to Connectivity > SIP Trunks in the admin panel.
  2. Click Add Trunk.
  3. Enter a descriptive Name (e.g., "ClearlyIP Primary", "Twilio Outbound").
  4. Select an Authentication Type from the dropdown.
  5. Enter the provider's Host address (FQDN or IP).
  6. Set the Port (default is 5061 for SIP TLS).
  7. Fill in authentication-specific fields:
    • IP ACL: Enter allowed IP addresses or CIDR ranges
    • Username/Password: Enter the credentials provided by your carrier
    • Mutual TLS: Ensure certificates are configured at the system level
  8. Configure Codec Preferences if your provider requires a specific order.
  9. Set Max Channels to match your provider contract (default: 30).
  10. Optionally select a Failover Trunk for automatic rerouting on failure.
  11. Choose an Inbound Caller ID Mode (default: Passthrough).
  12. Click Save.

After saving, the trunk will appear in the trunk list with a status indicator. The PBX will begin registering with the provider (for Username/Password trunks) or establishing the TLS connection. Monitor the status for confirmation that the trunk is active.

Field Reference

SIPTrunkCreate (Request)

Fields used when creating or updating a SIP trunk.

Field Type Required Default Description
name string (1–255) Yes — Human-readable name for the trunk
auth_type enum Yes — One of: IP_ACL, USERNAME_PASSWORD, MUTUAL_TLS
host string (1–255) Yes — Provider SIP server hostname or IP address
port int (1–65535) No 5061 SIP port; defaults to 5061 (standard SIP TLS port)
username string (max 255) No — SIP authentication username (for USERNAME_PASSWORD auth)
password string (max 255) No — SIP authentication password; encrypted in DB, never returned by the API
ip_acl string No — IP access control list for IP_ACL auth (comma-separated IPs or CIDR ranges)
codec_preferences dict No — Ordered codec list (see Codec Configuration)
max_channels int (1–9999) No 30 Maximum number of concurrent calls on this trunk
transport enum No TLS One of: TLS, TCP, UDP. TLS is mandatory for production.
inbound_cid_mode enum No PASSTHROUGH How inbound caller ID is handled: PASSTHROUGH, OVERRIDE, DETECT
failover_trunk_id UUID No — ID of the backup trunk to use on failure
notes string No — Free-form notes for administrative use

SIPTrunkResponse (API Response)

The API returns all create fields plus the following system-managed fields.

Field Type Description
id UUID Unique identifier for the trunk
tenant_id UUID Tenant that owns this trunk (set automatically)
is_active boolean Whether the trunk is currently enabled
created_at datetime Timestamp when the trunk was created
updated_at datetime Timestamp of the last modification
Sensitive Fields Omitted
The password and encrypted_password fields are never included in API responses. This is a deliberate security measure. To verify or change a trunk password, submit a new value via the update endpoint.

Codec Configuration

The codec_preferences field accepts a dictionary that defines the ordered list of codecs the PBX should offer during call negotiation. The order matters: codecs listed first are preferred.

Codec Bandwidth Typical Use
G.711 (PCMU/PCMA) 64 kbps Standard toll quality; widely supported by all providers
G.722 64 kbps Wideband HD audio; recommended for internal and modern trunks
G.729 8 kbps Low-bandwidth; useful for constrained links (requires licensing)
Opus 6–510 kbps Adaptive; ideal for WebRTC endpoints and variable-bandwidth paths

If codec_preferences is omitted, the PBX uses its global default codec order. Most providers work well with G.711 (PCMU) as the primary codec. Consult your provider's documentation for recommended settings.

Channel Limits

The max_channels field controls how many concurrent calls can use this trunk simultaneously. This value should match the capacity you have contracted with your provider.

  • Default: 30 concurrent channels
  • Range: 1 to 9999
  • When the channel limit is reached, new outbound calls will fail over to the failover trunk (if configured) or return a busy signal
  • Inbound calls that exceed the limit are rejected with a SIP 503 response
  • Monitor channel utilization in the Dashboard to plan capacity upgrades
Capacity Planning Tip
A common rule of thumb is one channel per 3–4 extensions for typical office use. High-volume call centers may need a 1:1 ratio or higher. Review your call history reports in Call History to determine peak concurrent call counts.

Inbound Caller ID Modes

The inbound_cid_mode field controls how the platform handles the caller ID information received from the provider on inbound calls.

Mode Behavior
PASSTHROUGH The caller ID received from the provider is passed directly to the endpoint without modification. This is the default and recommended setting for most deployments.
OVERRIDE The inbound caller ID is replaced with a value configured in the Caller ID Rules. Useful when you need to normalize or rebrand incoming call presentation.
DETECT The platform performs CNAM (Caller Name) lookup to enrich the caller ID with a name when only a number is provided. This may add slight latency to inbound call setup.

Trunk Failover

The failover_trunk_id field links a trunk to a backup trunk. When the primary trunk is unavailable or has reached its channel limit, calls are automatically rerouted to the failover trunk.

  • Failover is triggered by: trunk registration failure, SIP timeout (no response), SIP 503 (service unavailable), or channel limit exceeded
  • The failover trunk must be a separate, active trunk within the same tenant
  • Failover chains are supported (Trunk A fails to B, B fails to C) but circular references are rejected by the API
  • Failover events are logged and visible in the Audit Logs
Recommended Setup
For production reliability, configure at least two trunks from different providers with failover linking. For example, set ClearlyIP as the primary trunk and Twilio as the failover. This ensures PSTN connectivity survives a single provider outage.

Provider Configuration Examples

The following are general guidance notes for configuring trunks with commonly used providers. Always refer to your provider's current documentation for the most accurate settings.

ClearlyIP

  • Auth type: IP_ACL or USERNAME_PASSWORD (both supported)
  • Host: Provided in your ClearlyIP portal (typically a regional SBC address)
  • Port: 5061 (SIP TLS)
  • Codecs: G.711 PCMU preferred; G.722 supported on most routes
  • Notes: ClearlyIP provides DID management and SMS through the same trunk. See DIDs for number configuration.

Twilio Elastic SIP Trunking

  • Auth type: IP_ACL or USERNAME_PASSWORD
  • Host: Your Twilio SIP domain (e.g., your-trunk.pstn.twilio.com)
  • Port: 5061 (SIP TLS)
  • Codecs: G.711 PCMU/PCMA; Opus supported on select routes
  • Notes: Configure your Twilio IP ACL or credential list in the Twilio console to match the settings here. Twilio supports both origination (inbound) and termination (outbound) on the same trunk.

Monitoring & Troubleshooting

Keeping trunks healthy is critical for uninterrupted PSTN connectivity. Use the following tools and techniques to monitor and diagnose trunk issues.

Trunk Status Indicators

The trunk list in the admin panel shows a real-time status for each trunk:

Status Meaning
Registered The trunk is authenticated with the provider and ready for calls
Unregistered The trunk is not currently authenticated; check credentials or network
Failed Registration or connection attempts have failed repeatedly
Disabled The trunk has been manually deactivated (is_active = false)

Troubleshooting Checklist

  1. Trunk shows Unregistered:
    • Verify the host and port are correct
    • Confirm the provider account is active and not suspended
    • For USERNAME_PASSWORD: re-enter the password to rule out encryption issues
    • For IP_ACL: verify your PBX public IP matches the provider's whitelist
  2. Calls fail with "403 Forbidden":
    • Authentication credentials may be incorrect
    • The provider may be rejecting your source IP
    • Check Audit Logs for detailed SIP error codes
  3. One-way audio or no audio:
    • Verify SRTP is negotiated (both sides must support it)
    • Check firewall rules for RTP port ranges (typically 16384–32768)
    • Ensure NAT traversal is configured correctly if the PBX is behind a firewall
  4. Calls drop after 30 seconds:
    • This typically indicates a SIP session timer mismatch
    • Verify the provider supports SIP Session Timers (RFC 4028)
    • Check for NAT timeout issues on intermediate firewalls
  5. Channel limit exceeded:
    • Increase max_channels if your provider contract supports more
    • Configure a failover trunk to absorb overflow traffic
    • Review Call History for peak usage patterns
SIP Traces
For advanced troubleshooting, administrators with the super_admin role can enable SIP trace capture on the FreeSWITCH engine. SIP traces show the full signaling exchange and are invaluable for diagnosing registration, authentication, and call routing issues. See Admin Ops for details.
