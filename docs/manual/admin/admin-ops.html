Admin Operations

System maintenance tasks, health monitoring, and operational procedures for platform administrators.

Overview

Admin Operations covers the day-to-day maintenance tasks that keep the Aspendora Connect running smoothly. This includes pushing configuration to desk phones, managing voicemail storage, clearing caches, reloading FreeSWITCH components, verifying database health, and monitoring system status. These operations are typically performed by MSP administrators or tenant-level system administrators.

Role Requirement
Most admin operations require the System Admin or MSP Admin role. Tenant administrators can perform tenant-scoped operations but cannot affect other tenants or global system settings.

Phone Resync

When desk phone configuration changes are made (extension assignments, button mappings, firmware updates, or phonebook changes), those changes must be pushed to the physical devices. The resync operation sends a SIP NOTIFY message to the phone, instructing it to re-download its configuration from the provisioning server.

Resync a Single Phone

Navigate to the extension's detail page and click Resync Phone. The phone will reboot or re-fetch its configuration within a few seconds, depending on the phone model and firmware.

Bulk Resync

To resync all phones for a tenant, go to Admin Ops → Phone Resync and click Resync All. This sends NOTIFY messages to every registered device. Use this after making bulk changes such as updating a global phonebook or changing dial plan rules.

Phone Interruption
Resyncing phones may cause a brief interruption (1-3 seconds) as the device reboots or reloads configuration. Avoid bulk resyncs during peak call hours. Some phone models support hot-reload without a full reboot.

Voicemail Cleanup

Voicemail messages consume storage space in MinIO/S3. Over time, old messages can accumulate and increase storage costs. The voicemail cleanup operation purges messages that meet specified age criteria.

Cleanup Policy

Setting Default Description
Retention period (read) 90 days Delete read voicemails older than this
Retention period (unread) 180 days Delete unread voicemails older than this
Max messages per mailbox 200 Oldest messages removed when exceeded
Auto-cleanup schedule Daily at 02:00 UTC Automatic cleanup runs on this schedule

Manual Cleanup

To run a manual cleanup, go to Admin Ops → Voicemail Cleanup. You can preview which messages will be affected before confirming the operation. Deleted voicemails are permanently removed from object storage and cannot be recovered.

Cache Management

The platform uses Redis for caching frequently accessed data such as extension lookups, dial plan rules, tenant configuration, and session tokens. In rare cases, cached data may become stale and need to be manually cleared.

Cache Categories

Cache TTL Description
tenant:config 5 min Tenant settings and feature flags
ext:lookup 2 min Extension number-to-UUID resolution
dialplan:routes 5 min Compiled outbound routing rules
user:session 30 min Active user session metadata
did:routing 5 min Inbound DID-to-destination mapping

Clearing the Cache

Navigate to Admin Ops → Cache Management to flush specific cache categories or clear all caches. Clearing a cache causes the next request to rebuild it from the database, which may briefly increase response times.

When to Clear Cache
You should rarely need to clear caches manually. The platform automatically invalidates cache entries when underlying data changes. Manual clearing is useful if you suspect stale data after a database restore or manual SQL edits.

FreeSWITCH Management

FreeSWITCH is the media engine powering all voice calls. Some configuration changes require reloading FreeSWITCH modules without restarting the entire service. These operations are available to MSP admins only.

Reload Dialplan

After modifying dial plan XML templates, reload the dialplan module so FreeSWITCH picks up the new rules. Active calls are not affected by a dialplan reload. 

POST /api/v1/admin/freeswitch/reload-dialplan

Reload Sofia Profiles

Sofia is the SIP stack in FreeSWITCH. Reloading a Sofia profile applies changes to SIP settings such as TLS certificates, codec preferences, or NAT traversal. This operation gracefully updates the profile without dropping active calls. 

POST /api/v1/admin/freeswitch/reload-sofia

Gateway Status

View the registration status of SIP trunk gateways. A gateway shows as UP when registered with the carrier, DOWN when unreachable, or TRYING when a registration attempt is in progress.

Active Call Warning
Reloading Sofia profiles during high call volume may cause brief registration flaps on some SIP trunks. Schedule profile reloads during low-traffic windows when possible.

Database Maintenance

PostgreSQL is the primary data store. The platform uses Row-Level Security (RLS) to isolate tenant data. Regular verification ensures RLS policies are functioning correctly and that connection pools are healthy.

RLS Verification

The RLS verification check confirms that every tenant-scoped table has active RLS policies and that the application database user (new_phone_app) cannot access data across tenant boundaries. This check runs automatically daily and can be triggered manually. 

POST /api/v1/admin/database/verify-rls

Connection Pool Monitoring

The API server maintains a connection pool to PostgreSQL. Monitor pool utilization to detect connection leaks or capacity issues.

Metric Healthy Range Action if Exceeded
Pool size 5–20 connections Increase NP_DB_POOL_SIZE
Active connections < 80% of pool size Investigate slow queries
Waiting requests 0 Increase pool size or optimize queries
Overflow connections 0–5 Tune NP_DB_POOL_MAX_OVERFLOW

Health Checks

The platform exposes health check endpoints for monitoring systems and load balancers. Each subsystem reports its status independently, and the aggregate health endpoint returns an overall status.

Health Endpoint

GET /api/v1/health

Returns a JSON object with the status of each subsystem:

{
  "status": "healthy",
  "checks": {
    "api": "ok",
    "database": "ok",
    "redis": "ok",
    "freeswitch": "ok"
  },
  "version": "1.0.0",
  "uptime_seconds": 86400
}

Subsystem Checks

Subsystem Check Method Timeout Failure Means
API Self-check (always ok if responding) — API process is down
PostgreSQL SELECT 1 query 3 seconds Database unreachable or overloaded
Redis PING command 2 seconds Cache/pubsub layer down
FreeSWITCH Raw socket ESL connection 5 seconds Media engine unreachable
ESL Health Check
The FreeSWITCH health check uses a raw socket ESL connection rather than an ESL library. This approach was chosen because no reliable async ESL library is available on PyPI. The check connects, authenticates, sends api status, and disconnects.

System Status Monitoring

Beyond individual health checks, the platform provides system-level status monitoring for capacity planning and proactive issue detection.

Key Metrics

  • Active calls — Number of concurrent calls across all tenants
  • Registered endpoints — Count of SIP devices currently registered
  • API request rate — Requests per second to the API layer
  • API response time — P50, P95, and P99 latency percentiles
  • Queue depth — Number of callers waiting in all queues
  • Error rate — Percentage of 5xx responses
  • Disk usage — Object storage utilization for recordings and voicemail

Alert Thresholds

Configure alert thresholds to receive notifications when metrics exceed normal ranges. Alerts can be sent via email, webhook, or integrated with your existing monitoring stack.

Backup Procedures

Regular backups protect against data loss. The platform has three categories of data that require backup strategies.

Database Backups

PostgreSQL should be backed up using pg_dump or continuous archiving (WAL). The recommended approach is automated daily dumps with WAL archiving for point-in-time recovery.

Backup Type Frequency Retention Method
Full dump Daily 30 days pg_dump --format=custom
WAL archiving Continuous 7 days PostgreSQL archive_command
Configuration On change Indefinite Git-versioned config files

Media Backups

Call recordings and voicemail messages are stored in MinIO/S3. Use bucket replication or scheduled mc mirror to back up media files to a secondary location.

FreeSWITCH Configuration

FreeSWITCH XML configuration files, dialplan templates, and Sofia profiles should be version-controlled. The platform generates these from the database, but keeping a backup of the generated output aids in disaster recovery.

Troubleshooting Common Issues

The following table covers frequently encountered issues and their resolutions.

Issue Symptoms Resolution
Phone not registering Extension shows offline, phone display shows "No Service" Verify network connectivity, check SIP TLS certificates, resync phone configuration
One-way audio Caller can hear but callee cannot (or vice versa) Check NAT/firewall rules for RTP ports, verify SRTP configuration, check codec negotiation
API returning 503 Web UI shows connection errors, API health check fails Check API container logs, verify database connectivity, check connection pool exhaustion
Calls dropping after 30 seconds Calls connect but disconnect after exactly 30s SIP session timer issue; check FreeSWITCH session-expires settings and carrier compatibility
Voicemail not recording Callers hear greeting but message is not saved Check MinIO/S3 connectivity and bucket permissions, verify disk space
Cache stale data Changes made in UI not reflecting immediately Clear affected cache category from Admin Ops → Cache Management
High database latency API responses slow, connection pool warnings in logs Check PostgreSQL slow query log, run ANALYZE, verify index health
FreeSWITCH health check failing Health endpoint shows freeswitch: "error" Check if FreeSWITCH process is running, verify ESL port (8021) is accessible, check ESL password
Redis connection refused API startup fails, cache errors in logs Verify Redis container is running, check NP_REDIS_URL environment variable
Tenant isolation error RLS verification fails, cross-tenant data visible Immediately investigate; verify SET LOCAL app.current_tenant is being set per transaction
Tenant Isolation Failures
If RLS verification detects a cross-tenant data leak, treat this as a critical security incident. Immediately check the application database user permissions and ensure the app.current_tenant session variable is being set correctly in every database transaction.
