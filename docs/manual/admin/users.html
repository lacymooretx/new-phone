User Management

Create, edit, and manage user accounts, roles, and access within the platform.

Overview

Users are the people who log in to the Aspendora Connect. Every user belongs to exactly one tenant and is assigned a role that controls what they can see and do. User accounts are separate from extensions; a user may or may not have an associated phone extension depending on their role and responsibilities.

The platform supports five roles across two tiers: MSP roles (for service provider staff) and tenant roles (for customer organization staff). The role assigned to a user determines their permissions throughout the entire system.

MSP vs Tenant Roles

MSP roles (msp_super_admin, msp_tech) can manage all tenants and have cross-tenant visibility. They are intended for service provider staff only.

Tenant roles (tenant_admin, tenant_manager, tenant_user) are scoped to a single tenant. They cannot see or affect other tenants. Tenant admins manage their own organization; tenant users access only their own resources.

Who Can Manage Users

RoleCreate UsersEdit UsersAssign RolesDeactivateScope
msp_super_admin Yes Yes All roles Yes All tenants
msp_tech Yes Yes All roles Yes All tenants
tenant_admin Yes Yes Tenant roles only Yes Own tenant

Role Escalation Prevention
Tenant administrators cannot assign MSP-level roles. A tenant_admin can only assign tenant_admin, tenant_manager, or tenant_user roles within their own tenant. Only MSP roles can create or promote users to MSP roles.

Creating a User

To create a new user, submit a POST request to /api/v1/users with a JSON body matching the UserCreate schema.

  1. Navigate to System > Users in the admin interface.
  2. Click Add User.
  3. Fill in the required fields: email, password, first name, and last name.
  4. Select a role from the dropdown. The default is tenant_user.
  5. Click Save. The user account is created and active immediately.

UserCreate Field Reference

FieldTypeRequiredConstraintsDescription
email string Yes Max 320 characters, valid email format Login email address. Must be unique within the tenant.
password string Yes 8–128 characters Initial password. Hashed with bcrypt before storage. The user should change this on first login.
first_name string Yes 1–100 characters User's first name, displayed in the UI and directory.
last_name string Yes 1–100 characters User's last name, displayed in the UI and directory.
role UserRole enum No Default: tenant_user One of: msp_super_admin, msp_tech, tenant_admin, tenant_manager, tenant_user.

Editing a User

To update an existing user, send a PATCH request to /api/v1/users/{user_id} with a JSON body matching the UserUpdate schema. All fields are optional; only included fields will be modified.

The UserUpdate schema accepts:

FieldTypeDescription
emailstringUpdate the user's email address.
first_namestringUpdate first name.
last_namestringUpdate last name.
roleUserRole enumChange the user's role (subject to escalation rules).
is_activebooleanEnable or disable the account.

Assigning Roles

The platform uses a five-role RBAC model. Roles are assigned at user creation and can be changed later via the UserUpdate endpoint or the admin UI.

RoleTierDescription
msp_super_admin MSP Full platform access. Can manage all tenants, users, and system configuration.
msp_tech MSP Technical operations across all tenants. Can configure telephony, trunks, and troubleshoot.
tenant_admin Tenant Full administration within their tenant. Manages users, extensions, routing, and settings.
tenant_manager Tenant Operational management with configurable permissions. Can manage queues, reports, and optionally users.
tenant_user Tenant Standard user. Can access their own extension, voicemail, call history, and softphone.

For a full breakdown of permissions per role, see Roles & Permissions.

Enabling & Disabling Accounts

User accounts have an is_active flag that controls login access. Disabling an account is a soft action that preserves all user data and associations.

  • Active (is_active: true) — the user can log in and use the platform normally.
  • Disabled (is_active: false) — the user cannot log in. Active sessions are invalidated. Their extension (if any) remains configured but the user cannot access the softphone or web UI.

To re-enable a disabled account, set is_active: true via the update endpoint or toggle the status in the admin UI.

MFA Status

The mfa_enabled field on user records indicates whether the user has completed MFA enrollment. MFA is configured per-user; administrators can view MFA status but cannot enroll on behalf of a user.

  • mfa_enabled: true — the user has enrolled a TOTP authenticator, WebAuthn device, or SMS fallback.
  • mfa_enabled: false — the user has not yet set up MFA. If your tenant policy requires MFA, the user will be prompted on their next login.

For details on MFA methods, enrollment flows, and tenant-level enforcement policies, see Multi-Factor Authentication.

Password Management

Administrators set the initial password when creating a user. Passwords are hashed with bcrypt before storage; the platform never stores or displays plaintext passwords.

Password Rules

  • Minimum 8 characters, maximum 128 characters.
  • The platform does not currently enforce complexity rules (uppercase, symbols, etc.), but this may be configurable in a future release.
  • Users change their own password through the profile settings page or the POST /api/v1/auth/change-password endpoint.
  • Administrators cannot view existing passwords. To reset a forgotten password, an admin can issue a password reset or create a new temporary password.
SSO Users
Users who authenticate via Microsoft Entra ID or Google Workspace SSO do not use platform-managed passwords. Their password field is not applicable, and authentication is delegated to the identity provider. See Authentication for SSO setup.

Bulk User Operations

For onboarding larger organizations, the platform supports bulk operations through the API. Common bulk workflows include:

  • Bulk create — submit an array of UserCreate objects to POST /api/v1/users/bulk. Each user is validated independently; failures for individual records do not roll back the entire batch.
  • Bulk deactivate — submit an array of user IDs to POST /api/v1/users/bulk-deactivate. Useful when offboarding a department or group.
  • CSV import — the admin UI provides a CSV import tool under System > Users > Import. Upload a CSV with columns: email, first_name, last_name, role, password.

Listing & Filtering Users

Retrieve users with GET /api/v1/users. The endpoint supports pagination and filtering via query parameters:

ParameterTypeDescription
pageintegerPage number (default: 1).
per_pageintegerResults per page (default: 25, max: 100).
rolestringFilter by role (e.g., ?role=tenant_admin).
is_activebooleanFilter by account status (true or false).
searchstringSearch across email, first name, and last name.
sortstringSort field: email, last_name, created_at, last_login_at.
orderstringSort direction: asc or desc (default: asc).

Examples

Create a User

POST /api/v1/users
Content-Type: application/json
Authorization: Bearer <token>

{
  "email": "jane.doe@acmecorp.com",
  "password": "SecurePass123!",
  "first_name": "Jane",
  "last_name": "Doe",
  "role": "tenant_admin"
}

Update a User

PATCH /api/v1/users/<user_id>
Content-Type: application/json
Authorization: Bearer <token>

{
  "role": "tenant_manager",
  "last_name": "Smith"
}

Disable an Account

PATCH /api/v1/users/<user_id>
Content-Type: application/json
Authorization: Bearer <token>

{
  "is_active": false
}

List Active Tenant Admins

GET /api/v1/users?role=tenant_admin&is_active=true&sort=last_name&order=asc
Authorization: Bearer <token>
