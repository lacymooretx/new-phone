Inbound Routes

Control how incoming calls reach their destination by mapping DIDs to extensions, queues, IVR menus, and other endpoints.

Overview

An inbound route is the bridge between a phone number (DID) arriving on a SIP trunk and the internal destination that should handle the call. When an external caller dials one of your numbers, the platform looks up the matching inbound route, applies any caller ID modifications, evaluates time conditions, and delivers the call to the configured destination.

Without an inbound route, incoming calls on a DID have nowhere to go and will be rejected. Every DID that should accept calls needs at least one inbound route pointing it to a destination.

Route Per DID
Each DID can be associated with one inbound route. If you need different behavior at different times of day, use a single route that points to a Time Condition, which then branches to multiple destinations based on schedule.

Call Flow Diagram

The following diagram illustrates the path an inbound call takes through the routing engine.

PSTN Caller │ ▼ SIP Trunk (receives call for DID +15551234567) │ ▼ Inbound Route Lookup (match by DID) │ ├─ CID name prefix applied (e.g. "Main: ") │ ├─ Time conditions evaluated? │ ├─ Yes → Route to time-based destination │ └─ No → Continue to default destination │ ▼ Destination ├─ Extension (desk phone / softphone) ├─ Ring Group (multiple phones ring) ├─ Queue (ACD distribution to agents) ├─ IVR Menu ("Press 1 for sales...") ├─ Conference Room ├─ Voicemail Box └─ Time Condition (schedule-based branching)

Creating an Inbound Route

To create a new inbound route, submit a POST request to the inbound routes endpoint with the fields described below. At minimum, you must provide a name and a destination type. Linking a DID is optional at creation time but required for the route to actually receive calls.

Example: Route Main Number to IVR

POST /api/v1/inbound-routes

{
  "name": "Main Office Line",
  "did_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "destination_type": "IVR",
  "destination_id": "f9e8d7c6-b5a4-3210-fedc-ba9876543210",
  "cid_name_prefix": "Main: ",
  "enabled": true
}

Example: Route Direct Line to Extension

POST /api/v1/inbound-routes

{
  "name": "John Smith Direct",
  "did_id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
  "destination_type": "EXTENSION",
  "destination_id": "c3d4e5f6-a7b8-9012-cdef-123456789012",
  "enabled": true
}

Field Reference

InboundRouteCreate Fields

Field Type Required Default Description
name string Yes — Descriptive name for the route (1–255 characters)
did_id UUID No null Links this route to a specific DID number. The route will only receive calls arriving on this DID.
destination_type enum Yes — The type of destination. One of: EXTENSION, RING_GROUP, QUEUE, IVR, CONFERENCE, VOICEMAIL, TIME_CONDITION.
destination_id UUID No null The UUID of the destination object (e.g., the extension ID, queue ID, or IVR ID). Required for all destination types except when routing to a catch-all.
cid_name_prefix string No null Text prepended to the caller ID name (max 50 characters). Example: "Sales: " causes callers to display as "Sales: John Doe".
time_conditions dict No null JSON object defining time-based routing rules. When set, the route evaluates these conditions before delivering to the default destination.
enabled boolean No true Whether the route is active. Disabled routes are skipped during call processing.

Response Fields

The API returns all create fields plus the following system-managed fields.

Field Type Description
id UUID Unique identifier for the inbound route
tenant_id UUID The tenant that owns this route (set automatically from auth context)
is_active boolean Computed field reflecting whether the route is both enabled and has a valid destination
created_at datetime Timestamp when the route was created
updated_at datetime Timestamp of the last modification

Destination Types

Each inbound route must specify a destination type that determines where the call is delivered. Choose the type that matches your use case.

Type Description When to Use
EXTENSION Routes the call directly to a single extension (desk phone, softphone, or mobile app). Direct dial numbers assigned to individual users. Example: an executive's private line.
RING_GROUP Rings a group of extensions simultaneously or sequentially. Small teams where any member can answer. Example: a front desk with three phones that all ring.
QUEUE Places the caller in an ACD queue with hold music and announcements while waiting for the next available agent. Call centers and support lines with formal agent distribution. Example: a technical support hotline.
IVR Plays an auto-attendant menu and routes based on caller key presses. Main company numbers that need to offer choices. Example: "Press 1 for sales, 2 for support."
CONFERENCE Sends the caller directly into a conference bridge. Dedicated conference dial-in numbers. Example: a recurring meeting bridge.
VOICEMAIL Routes the call directly to a voicemail box without ringing any phone. After-hours numbers or informational lines. Example: a fax-back line that goes straight to voicemail.
TIME_CONDITION Evaluates a time-of-day/day-of-week schedule and branches to different destinations. Numbers that need different behavior based on business hours. Example: IVR during the day, voicemail at night.

Caller ID Name Prefix

The cid_name_prefix field lets you prepend a short label to the caller's name as it appears on the phone display. This is especially useful when a user receives calls on multiple DIDs and needs to know which number the caller dialed.

Common Prefix Patterns

Prefix Caller Display Use Case
"Sales: " Sales: Jane Smith Identify calls arriving on the sales DID
"Main: " Main: Jane Smith Distinguish main line calls from direct dial
"After Hrs: " After Hrs: Jane Smith Flag calls arriving on the after-hours number
"800: " 800: Jane Smith Indicate the call came in on a toll-free number
Prefix Length
Keep prefixes short. The caller ID name field on most phones is limited to about 15 characters total. A long prefix will truncate the actual caller name.

Time Conditions Integration

Time conditions allow a single inbound route to deliver calls to different destinations based on the time of day, day of week, or specific dates. There are two ways to apply time-based routing:

Option 1: Route to a Time Condition Destination

Set destination_type to TIME_CONDITION and destination_id to the UUID of a pre-configured time condition object. The time condition itself defines the schedule and the branching destinations. This is the recommended approach because the time condition can be reused across multiple routes and managed independently.

Option 2: Inline Time Conditions

Set the time_conditions field on the route itself with a JSON object defining schedule rules. This approach is simpler for one-off configurations but does not allow reuse. The inline conditions are evaluated before the default destination. 

{
  "time_conditions": {
    "business_hours": {
      "days": ["mon", "tue", "wed", "thu", "fri"],
      "start_time": "08:00",
      "end_time": "17:00",
      "destination_type": "IVR",
      "destination_id": "aaaabbbb-cccc-dddd-eeee-ffffffffffff"
    },
    "after_hours": {
      "destination_type": "VOICEMAIL",
      "destination_id": "11112222-3333-4444-5555-666677778888"
    }
  }
}
Time Zone Awareness
Time conditions use the tenant's configured time zone. Make sure the tenant time zone is set correctly under Tenant settings, or schedule-based routing will evaluate against the wrong clock.

Route Priority and Matching

When an inbound call arrives, the routing engine matches it to a route using the following logic:

  1. DID match — The engine looks for a route whose did_id matches the called number. This is the primary matching criterion.
  2. Enabled check — Only routes with enabled: true are considered. Disabled routes are skipped.
  3. First match wins — If multiple routes match the same DID (which should not happen in normal configuration), the first match by creation date is used.
  4. No match — If no enabled route matches the DID, the call is rejected with a 404 (number not in service) response to the carrier.
One Route Per DID
The system enforces a unique constraint: each DID can be linked to only one inbound route at a time. Attempting to create a second route for the same DID will return a validation error.

Configuration Examples

Main Office Number to IVR with Business Hours

Route the company's main number to an IVR during business hours and to a voicemail box after hours. Uses a Time Condition destination to handle the branching. 

{
  "name": "Main Office Number",
  "did_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "destination_type": "TIME_CONDITION",
  "destination_id": "d4e5f6a7-b8c9-0123-defg-456789012345",
  "cid_name_prefix": "Main: ",
  "enabled": true
}

Direct Line to Extension

A direct inward dial (DID) number assigned to a specific user. Calls go straight to their extension with no menu or queue in between. 

{
  "name": "Sarah Johnson Direct",
  "did_id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
  "destination_type": "EXTENSION",
  "destination_id": "c3d4e5f6-a7b8-9012-cdef-123456789012",
  "enabled": true
}

Support Number to Queue

Route the support hotline to an ACD queue where callers wait for the next available agent. The CID prefix helps agents know the call came from the support line. 

{
  "name": "Support Hotline",
  "did_id": "e5f6a7b8-c9d0-1234-efab-567890123456",
  "destination_type": "QUEUE",
  "destination_id": "f6a7b8c9-d0e1-2345-fabc-678901234567",
  "cid_name_prefix": "Support: ",
  "enabled": true
}

Conference Bridge Dial-In

A dedicated number that sends callers directly into a conference room. 

{
  "name": "Weekly Standup Bridge",
  "did_id": "f6a7b8c9-d0e1-2345-fabc-678901234567",
  "destination_type": "CONFERENCE",
  "destination_id": "a7b8c9d0-e1f2-3456-abcd-789012345678",
  "enabled": true
}
