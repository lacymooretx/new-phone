Authentication & Login

How users authenticate with the platform, including password login, SSO, MFA challenges, and token lifecycle.

Overview

The New Phone Platform uses a token-based authentication system built on JSON Web Tokens (JWT). All API access requires a valid access token, which is obtained by authenticating with email and password credentials. If Multi-Factor Authentication (MFA) is enabled on the account, a secondary challenge step is required before tokens are issued.

API-First Authentication
All authentication flows go through the REST API. The web UI and mobile apps use the same endpoints documented here. All communication is encrypted over TLS.

Login Flow

To log in, a user submits their email address and password to the login endpoint. The API validates the credentials and returns one of two responses depending on whether MFA is enabled.

LoginRequest Schema

Field Type Required Description
email string Yes The user's email address
password string Yes The user's password (min 8, max 128 characters)

TokenResponse Schema

Returned on successful authentication (no MFA, or after MFA verification).

Field Type Description
access_token string JWT access token for API requests
refresh_token string Long-lived token used to obtain new access tokens
token_type "bearer" Always "bearer" — include in the Authorization header

Authentication Flow Diagram

The following diagram shows the complete authentication flow, including the MFA challenge branch.

User → Login (email/password) → API ├─ No MFA → TokenResponse (access_token, refresh_token) └─ MFA Enabled → MFAChallengeResponse (mfa_required, mfa_token) → User enters TOTP code → MFAChallengeRequest (mfa_token, code) → API → TokenResponse (access_token, refresh_token)

When MFA is not enabled, the user receives tokens immediately. When MFA is enabled, the API returns an MFAChallengeResponse instead, and the user must complete the second factor before tokens are issued.

MFA Challenge Flow

When a user with MFA enabled submits valid credentials, the API does not return tokens. Instead, it returns an MFA challenge that must be completed.

MFAChallengeResponse Schema

Field Type Description
mfa_required boolean Always true when MFA is needed
mfa_token string Temporary token that authorizes the MFA verification step

MFAChallengeRequest Schema

Submit the MFA token along with the user's TOTP code to complete authentication.

Field Type Required Description
mfa_token string Yes The temporary token from MFAChallengeResponse
code string Yes 6-digit TOTP code from the user's authenticator app

If the code is valid, the API returns a TokenResponse with access and refresh tokens. If the code is invalid or the mfa_token has expired, the request is rejected and the user must start the login process again.

Single Sign-On (SSO) Coming Soon

The platform will support enterprise SSO through two providers. SSO allows users to authenticate using their existing corporate identity without managing a separate password.

Microsoft Entra ID

Integration via both OIDC and SAML protocols. Tenant administrators will be able to configure their Entra ID tenant connection in the admin panel, mapping Entra groups to platform roles.

  • Protocol support: OpenID Connect (OIDC) and SAML 2.0
  • Automatic user provisioning on first login
  • Group-to-role mapping
  • Support for Conditional Access policies

Google Workspace

Integration via OIDC. Similar configuration flow to Entra ID, with Google Workspace organizational units mapped to platform roles.

  • Protocol support: OpenID Connect (OIDC)
  • Automatic user provisioning on first login
  • Organizational unit mapping
SSO Not Yet Available
SSO integration is planned for a future release. Currently, only email/password authentication is supported. Contact your MSP administrator for timeline updates.

Token Lifecycle

The platform uses a dual-token system with short-lived access tokens and long-lived refresh tokens.

Access Tokens

  • Short-lived JWT containing user identity, role, and tenant context
  • Included in every API request via the Authorization: Bearer <token> header
  • Automatically verified on each request; expired tokens are rejected with 401 Unauthorized
  • Contain claims: sub (user ID), tenant_id, role, exp (expiry), iat (issued at)

Refresh Tokens

  • Longer-lived token stored securely by the client
  • Used to request a new access token when the current one expires
  • Each refresh token can only be used once (rotation on use)
  • Revoked on logout or password change

Token Usage

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

The token_type field in the TokenResponse is always "bearer". Clients must include the access token in the Authorization header of every API request.

Session Management

Sessions are managed entirely through tokens. The platform does not use server-side session storage. Key behaviors include:

  • Login — Creates a new token pair (access + refresh)
  • Token refresh — Issues a new token pair, invalidates the old refresh token
  • Logout — Revokes the refresh token; the access token expires naturally
  • Password change — Revokes all active refresh tokens for the user
  • Admin revocation — Administrators can revoke all sessions for any user
Stateless Access Tokens
Access tokens are stateless JWTs verified by signature only. They cannot be individually revoked before expiry. This is why access token lifetimes are kept short. For immediate session termination, revoke the refresh token and wait for the access token to expire.

Password Requirements

All passwords must meet the following requirements:

Rule Value
Minimum length 8 characters
Maximum length 128 characters
Hashing algorithm bcrypt (cost factor 12)
Storage Only the bcrypt hash is stored; plaintext is never persisted

Passwords are validated at both the API schema level (length constraints) and during user creation or password change operations.

Account Lockout

To protect against brute-force attacks, the platform enforces account lockout after repeated failed login attempts.

  • After a configurable number of failed attempts, the account is temporarily locked
  • Locked accounts cannot authenticate until the lockout period expires
  • Failed attempt counters reset on successful login
  • Administrators can manually unlock accounts from the Users management page
  • Lockout events are recorded in the audit log
Rate Limiting
In addition to account lockout, the login endpoint is rate-limited by IP address. Excessive requests from a single IP will receive 429 Too Many Requests responses regardless of which account is being targeted.
