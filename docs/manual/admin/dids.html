DID Management

Manage Direct Inward Dial phone numbers assigned to your tenant, including E.164 formatting, provider tracking, and E911 designation.

Overview

DIDs (Direct Inward Dial numbers) are the phone numbers assigned to a tenant on the Aspendora Connect. Each DID represents an externally reachable number that can receive calls from the public telephone network (PSTN). DIDs are the entry points for inbound call routing — when someone dials one of your numbers, the platform matches it to an Inbound Route and delivers the call to the configured destination.

All phone numbers in the system must be stored in E.164 format, the international standard for phone number formatting. This ensures consistent handling across providers, regions, and features like caller ID, SMS, and E911.

What Is E.164?
E.164 is the ITU-T standard for international telephone numbering. Numbers start with a + followed by the country code and subscriber number, with no spaces, dashes, or parentheses. For example, a US number is formatted as +15551234567 (country code 1, then the 10-digit number).

Adding DIDs

Follow these steps to add a new DID to your tenant.

  1. Navigate to Connectivity > DIDs in the admin panel.
  2. Click Add DID.
  3. Enter the phone number in E.164 format (e.g., +15551234567).
  4. Select the Provider that supplies this number.
  5. Optionally enter the Provider SID (the provider's unique identifier for this number, useful for API reconciliation).
  6. Set the initial Status (defaults to ACTIVE).
  7. If this number is designated for emergency calls, enable Emergency DID.
  8. Click Save.

After saving, the DID will appear in your number inventory. To make the DID functional, you must create an Inbound Route that maps this number to a destination (extension, ring group, IVR, queue, etc.).

Bulk Import
If you are adding many numbers at once (e.g., during initial setup or a porting event), use the API's batch endpoint to create multiple DIDs in a single request. The web UI currently supports one-at-a-time creation.

E.164 Format

All phone numbers in the platform use E.164 format. The format requires a leading + followed by the country code and subscriber number, with no separators. The validation pattern is ^\+\d{1,19}$, allowing 1 to 19 digits after the +.

Country Example (Formatted) E.164 Country Code
United States (555) 123-4567 +15551234567 +1
United Kingdom 020 7123 4567 +442071234567 +44
Australia (02) 1234 5678 +61212345678 +61
Germany 030 12345678 +493012345678 +49
Canada (416) 555-7890 +14165557890 +1
Validation Is Strict
The API rejects numbers that do not match the E.164 pattern. Common mistakes include omitting the + prefix, including spaces or dashes, or using a leading zero after the country code. The web UI formats the number automatically when you enter digits, but API consumers must provide the exact E.164 string.

DID Providers

Each DID is tagged with a provider value that identifies where the number originates. This is used for inventory management, billing reconciliation, and provider-specific feature handling (e.g., SMS capabilities).

Provider Description Typical Use
CLEARLYIP Numbers provided through ClearlyIP Trunking Primary provider for most MSP deployments. Supports voice, SMS/MMS, and E911.
TWILIO Numbers provided through Twilio Secondary provider or primary for tenants needing Twilio-specific features. Supports voice, SMS/MMS.
INTERNAL Numbers used for internal routing only Virtual numbers not connected to the PSTN. Useful for inter-tenant routing or test environments.
OTHER Numbers from providers not listed above Catch-all for third-party or regional providers. Manually manage these numbers.

The provider_sid field stores the provider's unique identifier for the number. For Twilio, this is the Twilio SID (e.g., PN...). For ClearlyIP, this is the number ID from the ClearlyIP portal. This value is used for API-level reconciliation between the platform and the provider.

DID Status Lifecycle

Each DID has a status that tracks its current state in the number lifecycle. Status transitions reflect real-world events like provisioning, porting, and decommissioning.

PENDING ──> ACTIVE ──> INACTIVE │ │ │ └──> (re-activate) ──> ACTIVE │ └──> PORTED_OUT (terminal)
Status Meaning Can Receive Calls?
PENDING Number has been ordered or is in the process of being ported in. Not yet active on the provider side. Use this status during provisioning or while awaiting port completion. No
ACTIVE Number is live and can receive inbound calls. This is the normal operating status for all production DIDs. Yes
INACTIVE Number has been temporarily taken out of service. The number remains assigned to the tenant but will not route calls. Can be re-activated. No
PORTED_OUT Number has been ported to another provider or carrier. This is a terminal status — the number is no longer under the tenant's control. Retained for historical reference. No
Status vs. is_active
The status field tracks the business lifecycle of the number, while is_active is a system-level toggle. A number must have status = ACTIVE and is_active = true to receive calls. Setting is_active = false is a quick way to disable a number without changing its lifecycle status.

Emergency DIDs (E911)

DIDs marked with is_emergency = true are designated for Enhanced 911 (E911) service. When a user dials 911 from an extension associated with an emergency DID, the system routes the call through the E911 infrastructure and transmits the caller's registered location information (PIDF-LO) to the Public Safety Answering Point (PSAP).

  • At least one DID per tenant should be designated as an emergency DID
  • Emergency DIDs must be associated with a provider that supports E911 (CLEARLYIP and TWILIO both support it)
  • Location information is configured per-extension, not per-DID — the DID determines which number is sent as the callback number to the PSAP
  • The platform uses PIDF-LO (Presence Information Data Format — Location Object) to transmit granular location data including building, floor, and room
  • Regulatory compliance requires that E911 location data be kept current for all extensions, especially for remote workers
E911 Compliance Is Critical
Kari's Law and RAY BAUM's Act require multi-line telephone systems to support direct 911 dialing, notification to a designated security contact, and dispatchable location information. Ensure that every physical location has at least one emergency DID configured and that extension location data is accurate.

Linking DIDs to Inbound Routes

A DID by itself is just a number in your inventory. To make it functional, you must create an Inbound Route that tells the platform what to do when a call arrives on that number.

The typical workflow is:

  1. Add the DID to your tenant (this page)
  2. Navigate to Connectivity > Inbound Routes
  3. Create a new inbound route and select this DID as the match pattern
  4. Choose a destination: extension, ring group, queue, IVR menu, conference, or voicemail
  5. Optionally configure time-based routing using Time Conditions and Holiday Calendars

DIDs without an inbound route will not route calls — incoming calls to unrouted numbers receive a SIP 404 (Not Found) response.

One DID, Multiple Routes
You can create multiple inbound routes for the same DID with different time conditions. For example, route calls to a ring group during business hours and to a voicemail box after hours. See Inbound Routes for details.

Number Porting Coming Soon

Number porting allows you to transfer existing phone numbers from another carrier to the Aspendora Connect. The porting process is managed through the admin panel and coordinates with the winning and losing carriers to complete the transfer.

  • Submit a port request with the numbers to transfer and the current carrier's account information
  • Track port status through the admin panel (submitted, FOC received, scheduled, completed, rejected)
  • DIDs are created with status = PENDING during the port and automatically transition to ACTIVE upon completion
  • Port-out notifications alert administrators when a number is being ported away
Feature Not Yet Available
The number porting workflow is planned for a future release. Currently, DIDs must be manually added to the platform after they are provisioned with your provider. Contact your MSP administrator for porting assistance.

Self-Service Number Ordering Coming Soon

Self-service ordering will allow tenant administrators to search for and purchase new phone numbers directly from the admin panel, without contacting the MSP.

  • Search available numbers by area code, city, or prefix
  • Preview number availability from connected providers in real time
  • One-click provisioning — the number is purchased from the provider and added as an ACTIVE DID automatically
  • Optional approval workflows for MSPs that want to control number acquisition
  • Cost estimates displayed before purchase
Feature Not Yet Available
Self-service number ordering is planned for a future release. Currently, number ordering is handled through the provider's portal, and DIDs are manually added to the platform.
