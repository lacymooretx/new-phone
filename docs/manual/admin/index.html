Getting Started

Set up your Aspendora Connect instance from first login to first call.

Platform Overview

The Aspendora Connect is a multi-tenant, API-first PBX built for managed service providers. It replaces legacy systems like FreePBX and Asterisk with a modern, containerized architecture that separates the control plane from the media plane.

The control plane runs on FastAPI (Python) and handles all configuration, authentication, and business logic. The media plane runs on FreeSWITCH and handles call processing, conferencing, and audio. A Rust-based SIP proxy sits in front, enforcing TLS on all signaling and SRTP on all media.

Every tenant is isolated at the database level using PostgreSQL Row-Level Security. A single deployment can serve approximately 50 tenants, each with their own extensions, trunks, routing rules, and users.

Security by Default
All SIP signaling uses TLS and all media uses SRTP. Unencrypted SIP (UDP/TCP) and unencrypted RTP are not supported and cannot be enabled. This is a deliberate design choice.

Quick Start Checklist

Follow these steps in order to go from a fresh deployment to a working phone system. Each step links to the detailed documentation page.

  1. Log in to the web interface
    Navigate to your instance URL and sign in with the MSP super admin credentials created during deployment. If SSO is configured, use your Microsoft Entra ID or Google Workspace account.
    Dashboard overview →
  2. Create a tenant
    Go to MSP > Tenants and create your first tenant. Provide a name, domain, and timezone. The tenant is the isolation boundary for all configuration.
    Tenant management →
  3. Add users
    Switch into the tenant context and navigate to System > Users. Create user accounts for the tenant admin and staff. Assign appropriate roles.
    User management →
  4. Create extensions
    Go to Telephony > Extensions and create SIP extensions. Each extension needs a number (e.g. 1001), a display name, and is assigned to a user. Extension credentials are auto-generated.
    Extension configuration →
  5. Configure a SIP trunk
    Navigate to Connectivity > SIP Trunks and add a trunk to your SIP provider (ClearlyIP, Twilio, or another carrier). Enter the provider's SIP host, authentication credentials, and codec preferences.
    SIP trunk setup →
  6. Add DID numbers
    Under Connectivity > DIDs, add your phone numbers. Associate each DID with the SIP trunk it arrives on. DIDs are assigned to inbound routes in the next step.
    DID management →
  7. Create inbound and outbound routes
    Set up Inbound Routes to direct incoming calls from each DID to an extension, ring group, queue, or IVR. Create Outbound Routes to define how dialed numbers are matched and sent out through trunks.
    Inbound routes → · Outbound routes →
  8. Test calls
    Register a SIP phone or use the WebRTC softphone to make internal calls between extensions. Then make an outbound call through the trunk to verify external connectivity. Check Call History to confirm CDRs are being generated.
You are up and running
Once you complete these steps, your tenant has working inbound and outbound calling. Continue configuring ring groups, queues, IVR menus, voicemail, and other features as needed.

System Requirements

The platform runs entirely in Docker containers. The following are minimum requirements for a production deployment serving up to 50 tenants.

Component Minimum Recommended
CPU 4 cores 8+ cores
RAM 8 GB 16+ GB
Storage 100 GB SSD 500 GB+ NVMe (recordings)
OS Linux (any distro with Docker) Ubuntu 22.04+ / Debian 12+
Docker Docker Engine 24+ Docker Engine 25+
Docker Compose v2.20+ v2.24+
Network Public IP, ports 443, 5061, 10000-20000/UDP Dedicated host, low-latency link
TLS Certificate Valid certificate for SIP domain Wildcard or SAN cert
Port Requirements
SIP TLS uses port 5061. The RTP media range (10000-20000 UDP) must be open and directly reachable. NAT traversal is handled by the Rust SIP proxy, but the RTP ports must not be blocked by a firewall.

Role Overview

The platform uses a five-role RBAC model. Roles are hierarchical: higher roles inherit all permissions of lower roles. Each user is assigned exactly one role.

Role Scope Description
msp_super_admin Global Full access to all tenants and all system settings. Can create and delete tenants, manage MSP-level users, and access all administrative functions across the entire platform.
msp_tech Global Technical access across all tenants. Can configure trunks, extensions, routing, and telephony features for any tenant. Cannot create or delete tenants or manage MSP-level users.
tenant_admin Single Tenant Full administrative access within one tenant. Can manage users, extensions, trunks, routing, queues, IVR, and all tenant-level settings. Cannot access other tenants or MSP functions.
tenant_manager Single Tenant Manages day-to-day telephony within one tenant. Can view and edit extensions, ring groups, queues, and call reports. Cannot manage users, trunks, or tenant settings.
tenant_user Single Tenant End-user access only. Can view their own extension settings, voicemail, call history, and use the WebRTC softphone. Cannot modify system configuration.

For details on per-feature permission mappings and how to assign roles, see Users & Roles.

Authentication

The platform supports three authentication methods. All methods enforce multi-factor authentication.

Email + Password with MFA

Traditional login with email and password. After password verification, users must complete a second factor: TOTP (authenticator app), WebAuthn (hardware key or biometrics), or SMS fallback. MFA enrollment is mandatory on first login.

Microsoft Entra ID SSO

OpenID Connect integration with Microsoft Entra ID (formerly Azure AD). Users authenticate through Microsoft's login flow and are automatically provisioned in the platform on first sign-in. Configure the Entra ID application in Authentication.

Google Workspace SSO

OpenID Connect integration with Google Workspace. Users authenticate through Google's login flow. Requires configuring an OAuth 2.0 client in Google Cloud Console. Configure in Authentication.
