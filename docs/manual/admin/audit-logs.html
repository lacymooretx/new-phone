Audit Logs

Every administrative action on the platform is recorded in an immutable audit log. Use audit logs for security investigations, compliance reporting, and change tracking.

Overview

The audit log captures a tamper-evident record of every significant action performed on the platform. When a user creates an extension, modifies a queue, deletes a trunk, or logs into the system, the platform writes an audit log entry with the actor's identity, what changed, when it changed, and from where the request originated.

Audit logs are append-only and immutable. Once written, entries cannot be modified or deleted by any user, including MSP super-administrators. This guarantees a trustworthy record for compliance audits, security investigations, and operational troubleshooting.

Automatic and Comprehensive
Audit logging is always enabled and cannot be disabled. Every CRUD operation on every resource type generates an audit log entry automatically. No configuration is required to activate logging.

Audit Log Field Reference

The following table describes every field returned in an AuditLogResponse object.

Field Type Description
id UUID Unique identifier for this audit log entry
user_id UUID | null The user who performed the action. Null for system-initiated actions (automated processes, scheduled tasks, webhook-triggered operations). See System vs User Actions.
tenant_id UUID | null The tenant context of the action. Null for MSP-level operations that span multiple tenants (such as tenant creation or global configuration changes).
action string The type of operation performed: "create", "update", "delete", "login", "logout", and others. See Action Types.
resource_type string The kind of resource affected: "extension", "user", "queue", "trunk", and others. See Resource Types.
resource_id UUID | null The specific resource instance that was affected. Null for actions that do not target a single resource (such as login or bulk operations).
changes object | null A JSON object containing before/after values for each modified field. Null for non-update actions or when change tracking is not applicable. See Changes Field.
ip_address string The IP address from which the request originated. For API requests, this is the client's IP. For system actions, this may be the server's internal address.
user_agent string | null The User-Agent header from the HTTP request. Identifies the browser, application, or API client that made the request. Null for system-initiated actions.
created_at datetime Timestamp when the audit log entry was recorded (ISO 8601, UTC). This is the authoritative timestamp for when the action occurred.

What Gets Logged

The audit log captures all state-changing operations on all platform resources. This includes every create, update, and delete operation, as well as authentication events and administrative actions. The following categories are always logged:

  • Resource CRUD — Creating, updating, or deleting any telephony resource (extensions, queues, ring groups, IVRs, trunks, DIDs, routes, etc.)
  • User management — Creating, updating, disabling, or deleting user accounts. Changing roles or permissions.
  • Authentication events — Successful and failed login attempts, logouts, MFA enrollment and verification, password changes, token refresh, session revocation.
  • Tenant operations — Creating, updating, suspending, or deleting tenants. Changing tenant-level settings.
  • Recording access — Generating playback URLs, downloading recordings, deleting recordings. Provides a chain of custody for sensitive audio data.
  • Configuration changes — Modifying time conditions, holiday calendars, caller ID rules, audio prompts, outbound dial patterns, and other system settings.
  • Bulk operations — Batch imports, exports, and bulk update operations. Logged as a single entry with summary details.
Read Operations
Standard read/list operations (viewing a list of extensions, reading a CDR, etc.) are not logged in the audit trail to avoid excessive log volume. The exception is recording playback, which is logged due to the sensitive nature of call recordings.

Action Types

The action field categorizes what kind of operation was performed. The core CRUD actions are supplemented by authentication-specific and administrative action types.

Action Description Example
create A new resource was created. The changes field contains the initial values of all fields. Administrator creates a new extension 201.
update An existing resource was modified. The changes field shows before/after values for each changed field. Administrator changes the voicemail PIN on extension 201.
delete A resource was removed. The changes field may contain the final state of the resource before deletion. Administrator deletes an unused SIP trunk.
login A user successfully authenticated. The resource_type is "session" and resource_id is null. User john@example.com logs in from 192.168.1.50.
login_failed An authentication attempt failed. Captures the email used and the failure reason (bad password, locked account, etc.). Failed login attempt for jane@example.com from 10.0.0.5.
logout A user explicitly logged out, revoking their refresh token. User john@example.com logs out.
mfa_enroll A user enrolled in multi-factor authentication. User jane@example.com enables TOTP MFA.
mfa_verify A user completed an MFA challenge during login. User jane@example.com verifies TOTP code after password login.
password_change A user changed their password. All active sessions for the user are revoked. User john@example.com changes their password.
session_revoke An administrator forcibly revoked all sessions for a user. Admin revokes all sessions for a compromised account.
recording_access A user generated a playback URL or downloaded a call recording. Supervisor plays back a recording for quality review.

Resource Types

The resource_type field identifies what kind of platform object was affected. Combined with action and resource_id, it provides a complete picture of what was done to which object.

Resource Type Description
extension SIP extensions (phone lines assigned to users or devices)
user User accounts (authentication, roles, profile information)
queue Call queues (ACD queues with agent management)
ring_group Ring groups (simultaneous or sequential ringing patterns)
ivr IVR menus (interactive voice response auto-attendants)
trunk SIP trunks (connections to PSTN carriers)
did DIDs (phone numbers assigned to the tenant)
inbound_route Inbound routing rules (DID-to-destination mapping)
outbound_route Outbound routing rules (dial pattern-to-trunk mapping)
conference Conference rooms (multi-party calling)
page_group Paging groups (one-way broadcast to multiple devices)
time_condition Time-based routing conditions
holiday_calendar Holiday calendar definitions
caller_id_rule Caller ID manipulation rules
audio_prompt Custom audio prompts and hold music
recording Call recordings (access, deletion, legal hold changes)
tenant Tenant organizations (MSP-level operations)
session Authentication sessions (login, logout, token events)
follow_me Follow-me routing configurations

Changes Field

For update actions, the changes field contains a JSON object showing exactly what was modified. Each key in the object is a field name, and the value is an object with old and new properties representing the before and after states.

Structure

{
  "field_name": {
    "old": "previous value",
    "new": "updated value"
  },
  "another_field": {
    "old": 100,
    "new": 200
  }
}

Example: Extension Update

An administrator changes the caller ID name and enables call recording on extension 201. The changes field would contain: 

{
  "caller_id_name": {
    "old": "John Smith",
    "new": "John Smith - Sales"
  },
  "recording_policy": {
    "old": "never",
    "new": "always"
  }
}

Important Notes

  • Only modified fields appear in the changes object. Fields that were not changed are omitted.
  • For create actions, changes may contain the initial values with old set to null.
  • For delete actions, changes may contain the final values with new set to null.
  • Sensitive fields (passwords, MFA secrets, tokens) are never included in the changes object. They are replaced with "[redacted]".
  • Large values (such as audio file content) are truncated to prevent excessive log size.
Sensitive Data Redaction
The audit log never stores sensitive values. Password hashes, MFA secrets, API tokens, and other credential material are automatically redacted from the changes field. The log records that a password was changed but never stores the old or new value.

Filtering and Searching

The audit log list endpoint accepts filter parameters to help you find specific events. All filters are optional and can be combined with AND logic.

AuditLogListParams

Parameter Type Default Description
tenant_id UUID none Filter by tenant. MSP administrators can view logs across tenants; tenant administrators are restricted to their own tenant.
user_id UUID none Filter by the user who performed the action. Useful for investigating a specific user's activity.
action string none Filter by action type: "create", "update", "delete", "login", "login_failed", etc.
resource_type string none Filter by resource type: "extension", "user", "queue", "trunk", etc.
date_from datetime none Return entries created on or after this timestamp (ISO 8601, UTC)
date_to datetime none Return entries created on or before this timestamp (ISO 8601, UTC)
page integer 1 Page number for pagination (1-based). Combined with per_page to navigate through results.
per_page integer 50 Number of entries per page (1–200)

Common Filter Combinations

  • Security investigation — Set action to "login_failed" and date_from/date_to to a suspicious time window
  • User activity audit — Set user_id to the target user and leave other filters empty to see all their actions
  • Configuration changes — Set resource_type to the specific resource (e.g., "trunk") and action to "update"
  • Deletion review — Set action to "delete" to see all resource deletions in a period
  • Recording access trail — Set action to "recording_access" to see who listened to recordings

Pagination

Audit log results use page-based pagination (as opposed to the offset-based pagination used by CDRs and recordings). This model is more intuitive for browsing through sequential event logs.

  • First page: page=1&per_page=50 (default)
  • Second page: page=2&per_page=50
  • Custom page size: page=1&per_page=200 (maximum)

The maximum value for per_page is 200 entries per page. Results are returned in reverse chronological order (newest first) by default. Use date range filters to narrow the result set for large tenants with high log volume.

System vs User Actions

Most audit log entries have a user_id identifying the person who performed the action. However, some actions are performed by the system itself without direct human initiation. These system actions have a null user_id.

User-Initiated Actions

Any action triggered by an authenticated API request has a user_id value. This includes actions taken through the web UI, mobile app, or direct API calls. The ip_address and user_agent fields provide additional context about where and how the user made the request.

System-Initiated Actions

Actions triggered by automated processes have user_id set to null. Common system-initiated actions include:

  • Scheduled retention cleanup — Automatic deletion of expired recordings or CDRs based on retention policies
  • Account lockout — Automatic locking of accounts after too many failed login attempts
  • Token expiration — Automatic session cleanup when refresh tokens expire
  • Webhook-triggered operations — Actions initiated by external system integrations
  • Database maintenance — Automated data archival and optimization tasks

To distinguish system actions in the audit log, filter for entries where user_id is null. The ip_address for system actions will typically show an internal server address (e.g., 127.0.0.1 or the container IP).

Compliance

The audit log is designed to support common compliance frameworks used in telecommunications and managed services. Its immutability, comprehensive scope, and searchability make it suitable for the following regulatory and industry standards.

SOC 2

SOC 2 Type II audits require evidence of continuous monitoring and logging of system changes. The New Phone Platform audit log satisfies several Trust Service Criteria:

  • CC6.1 — Logical access controls are logged (login, logout, failed login, session revocation)
  • CC6.2 — Access to system components is restricted and tracked (role changes, permission modifications)
  • CC7.2 — Security events are monitored (failed logins, account lockouts, unusual access patterns)
  • CC8.1 — Changes to system components are authorized, tested, and documented (all CRUD operations logged with before/after values)

HIPAA Considerations

Healthcare organizations using the platform for patient communications must comply with HIPAA audit requirements. The audit log supports this through:

  • Recording access logging — every playback of a call recording is tracked with user identity, timestamp, and IP address
  • User access management — all changes to user accounts, roles, and permissions are logged
  • Immutable log storage — audit entries cannot be altered or deleted, satisfying the integrity requirement
  • Minimum retention of 6 years (configurable) for HIPAA-covered entities
Compliance Is Shared Responsibility
The audit log provides the technical controls and evidence needed for compliance, but compliance is a holistic process that includes organizational policies, personnel training, risk assessments, and more. The platform's audit capabilities are one component of your overall compliance program.

PCI-DSS

If your organization processes payment card data over the phone, the audit log supports PCI-DSS requirement 10 (track and monitor all access to network resources and cardholder data) through comprehensive logging of all administrative actions, access controls, and system changes.

Log Retention

Audit log entries are retained for a configurable period. The default and minimum retention periods depend on the compliance requirements of the tenant.

Tier Retention Storage Access
Active 1 year Primary database Full filtering, instant queries
Archive 3 years Compressed archive tables Date-range queries, slightly slower
Cold Storage 7 years Object storage (S3/MinIO) Manual retrieval, exported as JSON/CSV files

MSP administrators can extend retention periods per tenant. Common extended retention requirements include:

  • HIPAA — 6 years minimum
  • SOX — 7 years minimum
  • PCI-DSS — 1 year minimum, 3 years recommended
  • GDPR — Varies by purpose; ensure retention does not exceed what is necessary
Audit Logs Are Never Fully Deleted
Unlike other platform data, audit log entries are never permanently deleted during normal operation. Even when the retention policy window passes, entries transition to cold storage rather than being removed. This ensures a complete historical record is always available for legal and regulatory purposes.
