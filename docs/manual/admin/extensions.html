Extensions Configuration

Create, configure, and manage telephone extensions — the core telephony identity for every user and device on the platform.

Overview

Extensions are the core telephony entity in the New Phone Platform. Each extension represents a dialable number within the system (e.g., "100", "2001") and serves as the primary identity for phone registration, call routing, and caller ID presentation.

When an extension is created, the platform automatically generates a unique SIP username that physical phones and softphones use to register. Extensions can optionally be linked to a user account (for web/mobile access) and a voicemail box (for missed call handling).

Extensions carry their own caller ID settings, call forwarding rules, E911 location data, recording policies, and class-of-service restrictions. This makes each extension a self-contained telephony endpoint that can operate independently or as part of ring groups, queues, and IVR destinations.

Extensions vs. Users
Extensions and users are separate entities. A user is a login account; an extension is a phone line. A user may have zero or one linked extension, and an extension may exist without any linked user (e.g., a lobby phone or conference room phone). Link them via the user_id field on the extension.

Creating an Extension

To create a new extension, submit a POST request to /api/v1/extensions with a JSON body matching the ExtensionCreate schema, or use the admin UI:

  1. Navigate to Telephony > Extensions in the admin interface.
  2. Click Add Extension.
  3. Enter the extension number (e.g., "100"). This must be unique within the tenant.
  4. Optionally link a user account and voicemail box.
  5. Configure caller ID, call forwarding, E911 location, and other settings as needed.
  6. Click Save. The extension is created and a SIP username is automatically generated.

After creation, the extension appears in the extension list. The SIP username is available in the response and on the extension detail page. To provision a physical phone, you will need the SIP username and a SIP password (obtained via the password reset endpoint).

SIP Password Security
The SIP password is returned only once when you perform a password reset via POST /api/v1/extensions/{id}/reset-sip-password. It is not stored in retrievable form. Record it immediately or provision the phone at that time. If lost, reset again.

ExtensionCreate Field Reference

All fields available when creating an extension:

Field Type Required Constraints Description
extension_number string Yes 1–20 characters The dial number for this extension (e.g., "100", "2001"). Must be unique within the tenant.
user_id UUID No — Link this extension to a user account. The user gains softphone access and web/mobile control of the extension.
voicemail_box_id UUID No — Link to a voicemail box. Unanswered calls route to this box when no call forwarding destination is set.
internal_cid_name string No Max 100 characters Caller ID name shown on internal (on-net) calls between extensions.
internal_cid_number string No Max 20 characters Caller ID number shown on internal calls. Defaults to the extension number if not set.
external_cid_name string No Max 100 characters Caller ID name sent to the PSTN on outbound external calls.
external_cid_number string No Max 20 characters Caller ID number sent to the PSTN on outbound external calls. Must be a valid DID assigned to the tenant.
emergency_cid_number string No Max 20 characters Caller ID number sent when dialing emergency services (911/E911). Must match a DID registered with the E911 provider.
e911_street string No — Street address for E911 location (e.g., "123 Main St Suite 400").
e911_city string No — City for E911 location.
e911_state string No — State/province for E911 location (e.g., "TX", "CA").
e911_zip string No — ZIP/postal code for E911 location.
e911_country string No Max 2 characters, default "US" ISO 3166-1 alpha-2 country code for E911 location.
call_forward_unconditional string No Max 40 characters Destination for unconditional call forwarding. When set, all calls forward immediately without ringing the extension.
call_forward_busy string No Max 40 characters Destination when the extension is busy (all lines in use) or DND is enabled.
call_forward_no_answer string No Max 40 characters Destination when the extension rings for the configured ring time without answer.
call_forward_not_registered string No Max 40 characters Destination when no phone is registered to the extension (phone offline, network outage).
call_forward_ring_time integer No 5–120, default 25 Seconds to ring the extension before triggering the no-answer forward. Only applies when call_forward_no_answer is set.
dnd_enabled boolean No Default: false Do Not Disturb. When enabled, incoming calls are treated as if the extension is busy and route to the busy forward destination or voicemail.
call_waiting boolean No Default: true When enabled, a second incoming call beeps the phone while the first call is active. When disabled, a second call is treated as busy.
max_registrations integer No 1–10, default 3 Maximum number of SIP devices that can register simultaneously to this extension. All registered devices ring on incoming calls.
outbound_cid_mode enum No Default: INTERNAL Controls which caller ID is sent on outbound calls. See Caller ID Configuration for details.
class_of_service enum No Default: DOMESTIC Restricts outbound dialing permissions. See Class of Service.
recording_policy string No Default: "never" Call recording behavior: "never", "always", or "on_demand". See Recording Policies.
pickup_group string No Max 20 characters Assigns the extension to a call pickup group. Members of the same group can answer each other's ringing calls.
notes string No — Free-text notes for administrative reference (e.g., "Lobby phone, Building A").

Caller ID Configuration

Each extension has three sets of caller ID values, each used in a different context:

Caller ID Type Fields Used When
Internal internal_cid_name, internal_cid_number Calling another extension within the same tenant. Displayed on the receiving phone's screen.
External external_cid_name, external_cid_number Calling an outside number via a SIP trunk. The PSTN carrier delivers this to the called party.
Emergency emergency_cid_number Dialing 911 or other emergency numbers. Must be a DID registered with the E911 provider for proper PSAP routing.

Outbound CID Mode

The outbound_cid_mode field controls which caller ID is sent on outbound external calls:

Mode Behavior
INTERNAL (default) Uses the internal_cid_name and internal_cid_number for outbound calls. Useful when internal and external CID should match.
EXTERNAL Uses the external_cid_name and external_cid_number. Use this when the extension needs a specific outbound DID different from the internal number.
EXTENSION Sends the raw extension number as the outbound caller ID. Typically only useful for internal testing or SIP trunk configurations that override CID anyway.
STIR/SHAKEN Compliance
External caller ID numbers should be DIDs that your tenant owns and has registered with the SIP trunk provider. Using unregistered numbers may result in calls being flagged as spam or blocked under STIR/SHAKEN attestation rules. The platform validates external CID against the tenant's assigned DID inventory.

Call Forwarding

Extensions support four types of call forwarding, each triggered by a different condition. Forwarding destinations can be another extension number, an external phone number, a ring group, a queue, or any other dialable destination in the dial plan.

Type Field Trigger
Unconditional (CFU) call_forward_unconditional All incoming calls forward immediately. The phone does not ring. Takes priority over all other forwarding types.
Busy (CFB) call_forward_busy Triggers when the extension is busy (all lines occupied) or when DND is enabled.
No Answer (CFNA) call_forward_no_answer Triggers when the phone rings for call_forward_ring_time seconds without being answered.
Not Registered (CFNR) call_forward_not_registered Triggers when no SIP device is registered to the extension (phone powered off, network down, etc.).

Call Forwarding Flow

The following diagram shows how the platform evaluates call forwarding rules for an incoming call to an extension:

Incoming Call → Extension ├─ CFU set? → Forward unconditionally to CFU destination ├─ DND enabled? → CF Busy destination (or voicemail) ├─ Phone rings... │ ├─ Answered → Connected │ ├─ Busy (all lines occupied) → CF Busy destination │ ├─ No answer (ring_time elapsed) → CF No Answer destination │ └─ Not registered (phone offline) → CF Not Registered destination └─ No CF set, no answer → Voicemail (if linked)

Forwarding rules are evaluated in priority order: CFU first, then device state (not registered), then DND/busy, then ring timeout. If no forwarding destination is configured for the matching condition, the call falls through to voicemail (if a voicemail box is linked) or receives a busy signal.

Ring Time and No-Answer Forward
The call_forward_ring_time setting (default 25 seconds) controls how long the phone rings before the no-answer forward triggers. A typical ring cycle is about 6 seconds, so the default of 25 seconds gives approximately 4 ring cycles. Adjust this value based on user preference and workflow — for example, reception desks may want a longer ring time (45–60 seconds) while a sales queue overflow might use a shorter ring time (15 seconds).

E911 Location (PIDF-LO)

Each extension can have its own physical street address for Enhanced 911 (E911) purposes. When a user dials 911, the platform sends this location data to the Public Safety Answering Point (PSAP) using the PIDF-LO (Presence Information Data Format — Location Object) standard, ensuring that emergency responders are dispatched to the correct address.

E911 Address Fields

Field Example Description
e911_street 123 Main St Suite 400 Full street address including suite, floor, or unit number.
e911_city Austin City name.
e911_state TX Two-letter state or province code.
e911_zip 78701 ZIP or postal code.
e911_country US ISO 3166-1 alpha-2 country code. Defaults to "US".
E911 Compliance Is Critical

Federal regulations (Kari's Law and RAY BAUM's Act) require multi-line telephone systems to provide accurate dispatchable location information for every 911 call. For multi-floor or multi-suite offices, each extension should have its specific floor/suite in the e911_street field.

The emergency_cid_number must be a DID registered with your E911 service provider (e.g., ClearlyIP) and associated with the same address. Mismatched CID and address data can cause routing failures.

Remote and Mobile Workers

For users who work from home or travel, the E911 address on their extension should reflect their actual physical location. The admin UI and user self-service portal allow updating E911 addresses. Consider implementing a policy that requires remote workers to verify their E911 address quarterly or when they change locations.

Call Recording Policies

The recording_policy field on each extension controls whether calls are recorded. This is independent of any tenant-wide or route-level recording settings.

Policy Behavior
"never" (default) Calls are never recorded for this extension, regardless of other settings. Use for extensions where recording is prohibited (e.g., legal counsel lines).
"always" Every call involving this extension is automatically recorded. Both inbound and outbound calls are captured. Recordings are stored in S3/MinIO.
"on_demand" Recording is not automatic but can be started mid-call by the user pressing a feature code or using the softphone recording button.
Legal Considerations
Call recording laws vary by jurisdiction. Some states and countries require all-party consent (all participants must be informed), while others require only one-party consent. Consult your legal team before enabling "always" recording. The platform can be configured to play a recording announcement; see Audio Prompts for details.

Class of Service

The class_of_service field restricts what types of outbound calls an extension is permitted to make. This prevents unauthorized toll calls and controls international dialing access.

Class Allowed Destinations
EMERGENCY Emergency calls only (911). No local, long-distance, or international calls. Suitable for lobby phones or elevator phones.
DOMESTIC (default) Emergency calls plus all domestic calls (local and long-distance within the country). No international calls.
INTERNATIONAL Full dialing permissions including international calls. Assign only to extensions that require international access.

Class of service is enforced at the outbound route level. When an extension attempts a call that exceeds its class, the call is rejected with a "call restricted" announcement. The attempt is logged in the CDR with a disposition of RESTRICTED.

SIP Registration

Each extension is assigned an auto-generated sip_username when created. This username, combined with a SIP password, is what phones use to register with the platform over SIP TLS.

SIP Username

The SIP username is automatically generated by the platform and cannot be changed. It is returned in the extension response as sip_username and is visible on the extension detail page in the admin UI. The format is system-determined and guaranteed unique across the platform.

Max Registrations

The max_registrations field (default 3, range 1–10) controls how many SIP devices can simultaneously register with the same extension credentials. When multiple devices are registered, all devices ring on incoming calls (fork ringing).

Common configurations:

  • 1 registration — Single device only. Desk phone or softphone, not both.
  • 3 registrations (default) — Desk phone + softphone + mobile app. Covers the typical setup.
  • 5+ registrations — Shared extensions (e.g., warehouse paging phones, multiple devices at a reception desk).

SIP Password Reset

SIP passwords are managed separately from user account passwords. To set or reset the SIP password, call POST /api/v1/extensions/{id}/reset-sip-password. The response returns a SIPPasswordResetResponse:

Field Type Description
sip_password string The new SIP password. Returned only in this response — it cannot be retrieved later. Record it or provision the phone immediately.
SIP TLS Mandatory
All SIP registrations must use TLS. The platform does not accept SIP over UDP or TCP. When provisioning phones, ensure the transport is set to TLS and the SIP port is the platform's TLS port. SRTP is also mandatory for all media.

Do Not Disturb & Call Waiting

Do Not Disturb (DND)

When dnd_enabled is true, the extension is treated as busy for all incoming calls. Calls are routed to the call_forward_busy destination if configured, or to voicemail if a voicemail box is linked. The phone does not ring.

DND can be toggled by an administrator via the API or admin UI, or by the user via a feature code on their phone, the softphone interface, or the mobile app.

Call Waiting

When call_waiting is true (default), a second incoming call while the extension is on an active call produces a call-waiting beep on the phone. The user can switch between calls.

When call_waiting is false, a second incoming call while the extension is busy is treated as a busy condition and routes to the call_forward_busy destination or voicemail. This is useful for users who prefer not to be interrupted during calls.

Call Pickup Groups

The pickup_group field assigns an extension to a named pickup group. Any extension in the same pickup group can answer a ringing call on another extension in that group by dialing the directed call pickup feature code.

Pickup groups are identified by a string label (e.g., "sales", "support", "front-desk"). Extensions with the same pickup_group value are in the same group. An extension with no pickup_group set is not in any group and its calls cannot be picked up by others.

Example: Three extensions (100, 101, 102) all have pickup_group set to "reception". If extension 100 is ringing, a user at extension 101 or 102 can pick up that call by dialing the group pickup feature code.

Extension Response Fields

API responses for extensions include all fields from the create schema plus the following system-managed fields:

Field Type Description
id UUID Unique identifier for the extension record.
tenant_id UUID The tenant this extension belongs to. Set automatically; enforced by row-level security.
sip_username string Auto-generated SIP registration username. Used by phones to register with the platform.
agent_status string Current queue agent status (e.g., "available", "unavailable", "on_break"). Relevant when the extension is a member of a call queue.
is_active boolean true if the extension is active. Inactive extensions cannot register or receive calls.
created_at datetime Timestamp when the extension was created (UTC).
updated_at datetime Timestamp of the last modification (UTC).

API Examples

Create an Extension

POST /api/v1/extensions
Content-Type: application/json
Authorization: Bearer <token>

{
  "extension_number": "100",
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "internal_cid_name": "Jane Doe",
  "internal_cid_number": "100",
  "external_cid_name": "Acme Corp",
  "external_cid_number": "+15125551234",
  "emergency_cid_number": "+15125551234",
  "e911_street": "123 Main St Suite 400",
  "e911_city": "Austin",
  "e911_state": "TX",
  "e911_zip": "78701",
  "call_forward_no_answer": "200",
  "call_forward_ring_time": 30,
  "max_registrations": 3,
  "outbound_cid_mode": "EXTERNAL",
  "class_of_service": "DOMESTIC",
  "recording_policy": "on_demand",
  "pickup_group": "reception"
}

Reset SIP Password

POST /api/v1/extensions/a1b2c3d4-e5f6-7890-abcd-ef1234567890/reset-sip-password
Authorization: Bearer <token>

Response:
{
  "sip_password": "xK9#mP2$vL5nQ8wR"
}

Update Call Forwarding

PATCH /api/v1/extensions/a1b2c3d4-e5f6-7890-abcd-ef1234567890
Content-Type: application/json
Authorization: Bearer <token>

{
  "call_forward_unconditional": "+15125559999",
  "dnd_enabled": false
}
