Caller ID Rules

Match, block, replace, route, or tag inbound calls based on their caller ID number using pattern-based rules.

Overview

Caller ID rules allow administrators to take automated actions on inbound calls based on the calling party's number. Rules are evaluated in priority order against the caller ID of each incoming call, and the first matching rule's action is applied.

Common uses include blocking known spam callers, routing VIP customers to a priority queue, replacing malformed caller IDs with a normalized format, and tagging calls for reporting purposes.

Inbound Processing
Caller ID rules are evaluated early in the inbound call path, before the call reaches time conditions, IVR menus, or ring groups. A BLOCK action terminates the call immediately, while ROUTE, REPLACE, and TAG actions modify or redirect the call before it continues through the normal routing flow.

Creating Caller ID Rules

Navigate to System > Caller ID Rules and click Add Rule. Configure the match type, pattern, action, and priority. Rules take effect immediately after creation.

Take care when setting priorities. Rules are evaluated from lowest priority number to highest, and evaluation stops at the first match. Plan your priority assignments to ensure more specific rules are evaluated before general ones.

Field Reference

CallerIdRuleCreate Fields

Field Type Required Description
name string Yes Descriptive name (1–100 characters). Example: "Block 900 Numbers"
rule_type enum Yes Match type: PREFIX_MATCH, EXACT_MATCH, or REGEX
match_pattern string Yes The pattern to match against the caller ID (1–40 characters)
action enum Yes Action to take on match: BLOCK, REPLACE, ROUTE, or TAG
destination_id UUID No Target destination for the ROUTE action. Required when action is ROUTE
priority integer No Evaluation order (0–9999). Lower numbers are evaluated first. Default: 0
notes string No Optional notes explaining the purpose of this rule

Response Fields

The response includes all create fields plus the following system-managed fields:

Field Type Description
id UUID Unique identifier for the rule
tenant_id UUID Owning tenant (set automatically from auth context)
is_active boolean Whether the rule is currently active
created_at datetime Timestamp when the rule was created
updated_at datetime Timestamp of the last modification

Match Types

PREFIX_MATCH

Matches when the caller ID number starts with the specified pattern. This is the most commonly used match type, ideal for blocking area codes, country codes, or number ranges.

Pattern Matches Does Not Match
1900 19005551234, 19001234567 12125551234, 9001234567
1212 12125551234, 12129876543 12135551234, 2125551234
+44 +442071234567, +441234567890 12125551234, 442071234567

EXACT_MATCH

Matches only when the caller ID number is exactly equal to the pattern. Use this for specific known numbers, such as a VIP client's direct line or a known spam number.

Pattern Matches Does Not Match
12125551234 12125551234 12125551235, 2125551234
Anonymous Anonymous anonymous, ANONYMOUS

REGEX

Matches using a regular expression pattern. This provides the most flexibility but requires familiarity with regex syntax. The pattern is matched against the full caller ID string.

Pattern Purpose Matches
^1(900|976)\d+$ Premium rate numbers 19005551234, 19761234567
^(\+?1)?800\d{7}$ Toll-free 800 numbers 8005551234, 18005551234, +18005551234
^(Anonymous|Unavailable|Unknown)$ Blocked/private callers Anonymous, Unavailable, Unknown
Regex Performance
Complex regular expressions are evaluated on every inbound call. Keep patterns simple and test them thoroughly before deploying. Avoid unbounded quantifiers and backtracking-prone patterns that could impact call setup latency.

Actions

BLOCK

Immediately rejects the call. The caller hears a fast busy signal or a configurable rejection tone. The call is logged in CDRs with a "blocked" disposition.

  • Use case: Block known spam numbers, robocallers, or premium-rate prefixes
  • CDR impact: Call appears in call history with blocked status for audit purposes

REPLACE

Modifies the caller ID before the call continues through the routing flow. The match_pattern determines what to match, and the replacement value is derived from the pattern (for regex, capture groups can be used). The call then proceeds through normal inbound routing with the modified caller ID.

  • Use case: Normalize international formats, strip prefixes, or add country codes
  • CDR impact: Both original and modified caller ID are recorded

ROUTE

Overrides normal inbound routing and sends the call directly to a specific destination. Requires a destination_id pointing to an extension, ring group, queue, IVR, or other routable resource.

  • Use case: Route VIP clients to a priority queue, send known partners to a dedicated extension
  • CDR impact: Call is logged with the override destination noted

TAG

Adds a metadata tag to the call without changing its routing. The call proceeds through normal inbound routing, but the tag is available for reporting, queue prioritization, and CRM integration.

  • Use case: Tag internal calls, flag calls from specific regions, categorize callers for analytics
  • CDR impact: Tag is stored in the CDR record and available in reports

Priority and Evaluation Order

Rules are evaluated in ascending priority order (lowest number first). When a rule matches, its action is applied and no further rules are evaluated. If no rules match, the call proceeds through normal inbound routing unchanged.

Inbound Call (Caller ID: 12125559999) | v Rule priority 0: PREFIX_MATCH "1900" BLOCK -- no match, continue | v Rule priority 10: EXACT_MATCH "12125559999" ROUTE -- MATCH! Route to VIP queue | (stop evaluating) X (remaining rules skipped)

Recommended priority ranges:

Priority Range Suggested Use
0–99 Emergency blocks (spam, fraud, abuse)
100–499 VIP routing and exact-match rules
500–999 Prefix-based routing and replacements
1000–4999 General tagging and categorization
5000–9999 Catch-all and default rules
Tie-Breaking
When multiple rules share the same priority, their evaluation order among themselves is determined by creation time (oldest first). To ensure deterministic behavior, assign unique priorities to rules that could potentially match the same callers.

Common Use Cases

Block Spam Prefixes

{
  "name": "Block 900 Numbers",
  "rule_type": "PREFIX_MATCH",
  "match_pattern": "1900",
  "action": "BLOCK",
  "priority": 0,
  "notes": "Block premium-rate 900 numbers"
}

Route VIP Callers

{
  "name": "VIP - Acme Corp Main Line",
  "rule_type": "EXACT_MATCH",
  "match_pattern": "12125551000",
  "action": "ROUTE",
  "destination_id": "uuid-of-vip-queue",
  "priority": 100,
  "notes": "Route Acme Corp direct to VIP support queue"
}

Tag Internal Calls

{
  "name": "Tag Internal Extensions",
  "rule_type": "REGEX",
  "match_pattern": "^1(555)\\d{7}$",
  "action": "TAG",
  "priority": 1000,
  "notes": "Tag calls from our own DID range for reporting"
}

Block Anonymous Callers

{
  "name": "Block Anonymous/Unknown",
  "rule_type": "REGEX",
  "match_pattern": "^(Anonymous|Unavailable|Unknown|Restricted)$",
  "action": "BLOCK",
  "priority": 50,
  "notes": "Block callers hiding their caller ID"
}
